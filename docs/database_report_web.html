Basics

先把 Database 最基本的觀念講清楚

這一段是最重要的基礎。如果這些概念不穩，後面看 schema 或 table 名稱都只是背單字。

Database 是什麼

可以把 Database 想成一個有規則的資料倉庫。它不是單純把東西存起來，而是規定資料要怎麼分類、怎麼查、怎麼更新、怎麼避免重複和衝突。

Table 是什麼

Table 就像 Excel 的一張工作表，但更嚴格。每張 table 代表一種資料實體，例如使用者、收藏、通知、職缺。

Row 是什麼

Row 就是一筆資料。例如一個使用者、一筆收藏、一條通知。你可以把 row 想成「一張表格裡的一列」。

Column 是什麼

Column 是某種欄位，例如 id、email、created_at。同一個 table 的每筆 row 都會有同樣的欄位結構。

Primary Key 是什麼

Primary key 是每筆資料的唯一身份證。常見是 id。有了它，系統才知道你是在改哪一筆資料，而不是模糊地改到別筆。

Foreign Key 是什麼

Foreign key 是資料之間的關聯。例如收藏屬於哪個使用者，通常就會有 user_id 指向 users.id。它的目的，是避免出現「指到不存在的人」這種髒資料。

Index 是什麼

Index 像書的目錄。資料一多，如果沒有 index，查詢要慢慢一筆一筆掃。建立 index 後，常查的條件就能快很多。

Transaction 是什麼

Transaction 是「這一組操作要嘛全部成功，要嘛全部不算」。它用來避免只寫到一半就中斷，讓資料庫變成半完成、半失敗的狀態。

Schema 是什麼

Schema 就是整個資料結構設計圖。有哪些 table、每個 table 有哪些欄位、彼此怎麼關聯，這些加起來就是 schema。

為什麼不是直接用 JSON 就好

  • 少量資料、一次性輸出、前台直接讀取，JSON 很方便。
  • 但只要你需要更新某一部分、查詢某個條件、避免重複、建立關聯，Database 就更適合。
  • 所以 JSON 不是落後，Database 也不是萬能。要看資料的用途。

正規化是什麼

正規化就是把一大坨混在一起的資料拆成比較合理的表。簡單說，就是不要把本來應該一筆一筆存在 table 裡的資料，硬塞進一個超大的字串或 JSON 欄位裡。 

不理想：
saved_searches.known_job_urls = ["url1", "url2", "url3", ...]

較合理：
saved_search_seen_jobs
- search_id
- job_url
- ordinal
Core Ideas

你先記住這 5 件事，後面就不容易迷路

對初學者來說，先抓住主幹，比一次吸收全部細節更重要。

1. Database 主要處理「要長期留下來、還會再查的資料」

像 users、favorites、saved searches、notifications 這類產品狀態，就非常適合放 DB。

2. 不是所有資料都要放 DB

像最新報告快照、原始 HTML cache、暫時執行狀態，有時放 JSON 或檔案反而更合理。

3. 一個系統可以有不只一個資料檔

你這個專案就是這樣。不是只有一個 .sqlite3，而是不同用途拆開。

4. 架構好不好，不是看它多大，而是看它能不能維護

現在先做可維護的分層，比提早換大型 DB 更重要。

5. 真正危險的不是「資料多」，而是「資料亂」

如果沒有 key、index、relation、retention，就算資料還不大，也會開始難維護。

Project Map

把你這個專案當成 Database 教材來看

現在開始把上面的概念，直接套到你這個專案。這樣你學到的就不是抽象理論，而是你自己的系統。

data/cache

這不是正式業務資料庫，而是原始抓取結果的快取區。它的工作是幫系統省時間，不是當產品真相來源。

  • 特性：大、容易膨脹、可清理
  • 目的：減少重抓與重算
  • 不是：收藏、通知、使用者狀態的來源

jobs_latest.json

這是最新市場快照。它像公告板，告訴前台「目前最新結果是什麼」，而不是保存完整歷史。

  • 前台讀最新市場結果很方便
  • 目前有 34 jobs、22 skills、96 task insights
  • 適合展示，不適合承擔完整歷史分析

product_state.sqlite3

這是產品主庫。真正屬於產品本身的長期狀態，多半都在這裡。

  • users
  • saved_searches
  • saved_search_seen_jobs
  • favorite_jobs / notifications

user_submissions.sqlite3

這是履歷和提交資料的獨立庫。把它跟主庫拆開，是個很好的決定，因為這類資料未來常常有獨立清理、隱私或審計需求。

query_runtime.sqlite3 + data/snapshots/

這一層不是產品資料，而是爬蟲執行期協調層。它負責排隊、去重、保存 partial 或 final snapshot 檔案。

market_history.sqlite3

這是歷史分析層。程式碼已經做好了，它會在 finalized snapshot 後接手保存歷史 run 和對應 job 資料。

一句話理解這個分層

  • cache：省時間
  • jobs_latest.json：給前台看最新結果
  • product_state.sqlite3：保存產品狀態
  • user_submissions.sqlite3：保存提交資料
  • query_runtime.sqlite3：協調爬蟲工作
  • market_history.sqlite3：保存歷史分析資料

這樣拆的好處

  • 前台最新結果不用每次都從大表慢慢算。
  • 產品主庫不會被爬蟲中間資料污染。
  • 敏感資料可以獨立管理。
  • 之後若要擴充歷史分析，可以只擴 history layer，不必整個系統重做。
Flow

從使用者搜尋到資料落地，整個流程怎麼走

真正看懂 DB 系統，不是只看有哪些表，而是知道資料什麼時候被寫進哪裡。

1

使用者輸入查詢，系統先產生 query signature

這個 signature 是查詢的身份。未來 runtime queue、snapshot 命名、去重判斷都會靠它，而不是只看搜尋名稱。

目的：識別同一種查詢 避免重複抓取
2

query runtime 先判斷：這個查詢要不要真的再跑一次

系統會去看 query_runtime.sqlite3 裡是否已經有新鮮 snapshot，或是否已有 pending / running job。如果有，就不一定需要再抓。

表：query_snapshots 表：crawl_jobs
3

真正爬資料時，原始內容先進 cache

抓回來的原始頁面先進 data/cache。這樣下次相同或相似請求，不用每次都從頭打外部網站。

屬性：快取層 可清理，不是主真相
4

pipeline 把抓回來的資料整理成 market snapshot

這時候資料會變成比較有結構的成果，例如 jobs、skills、task insights、errors。也就是開始從「原始抓取資料」變成「產品能理解的市場資料」。

輸出：市場快照 可能落成 partial snapshot
5

完成後，最新結果會覆寫到 jobs_latest.json

這份檔案就是「現在前台看到的最新市場結果」。它適合拿來展示目前畫面，不是拿來做全部歷史分析。

用途：最新頁面 不是：完整歷史庫
6

同一份 finalized snapshot 也會寫進 history store

如果走到 final snapshot，系統會把它轉成 crawl_runs、job_posts、crawl_run_jobs 這些歷史資料。這是為了之後做分析，不是為了即時前台查詢。

DB：market_history.sqlite3 有 retention：90 天 / 每 query 100 runs
7

如果使用者有 saved search，就用主庫比對新舊職缺

系統會把本次看到的職缺 URL 與 saved_search_seen_jobs 做比對，判斷哪些是新職缺，再決定是否建立通知。

核心：去重 不再靠大 JSON 欄位
8

不同功能讀不同層，不是所有畫面都打主庫

產品收藏與通知讀主庫，最新市場畫面讀最新 snapshot，提交資料讀提交庫，歷史分析未來則讀 history store。這就是分層系統的價值。

最新畫面：JSON 產品狀態：SQLite 歷史分析：history DB
Tables

用初學者聽得懂的方式解讀目前的核心資料表

先不要把資料表想成技術名詞，把它想成「這張表想回答什麼問題」。

users

回答的問題是：系統裡有哪些使用者？

  • 每個 user 通常會有自己的 id
  • 其他表常常會透過 user_id 連到它

saved_searches

回答的問題是：某個使用者保存了哪些搜尋條件？

  • 這裡存的是搜尋設定本身
  • 像名稱、條件、signature、最後執行狀態

saved_search_seen_jobs

回答的問題是：這個 saved search 以前看過哪些 job URL？

  • 這張表是很重要的正規化改進
  • 避免把一大串 URL 硬塞在同一欄
  • 之後同步、裁切、查詢都比較乾淨

favorite_jobs

回答的問題是：某個使用者收藏了哪些工作？

  • 它通常至少會有 user_id 和 job 識別資訊
  • 這種表很典型，屬於產品狀態表

notification_preferences / job_notifications

回答的問題是：使用者希望怎麼被通知？系統實際產生過哪些通知？

  • 一張表偏設定
  • 一張表偏事件紀錄

user_submissions

回答的問題是：使用者提交過哪些履歷或資料？

  • 獨立一個 DB 很合理
  • 這類資料常常跟隱私和清理策略有關

query_snapshots / crawl_jobs

回答的問題是：這個查詢現在有沒有現成結果？有沒有已經排進隊列的工作？

  • 這是執行期狀態，不是產品真相
  • 空表不代表設計沒用

crawl_runs / job_posts / crawl_run_jobs

回答的問題是：過去 crawl 過幾次？每次抓到什麼？某個 job 在歷史上出現過嗎？

  • 這是分析層骨架
  • 不是前台最新畫面的主要來源

你可以怎麼分辨一張表該不該存在

  • 如果這是一種穩定的資料實體，它通常值得是一張表。
  • 如果這筆資料未來要被查詢、統計、更新，它更值得是一張表。
  • 如果這只是一次性輸出、只是快照、只是暫存，不一定要是表。

為什麼 saved_search_seen_jobs 是好例子

它很適合拿來學 DB。因為你可以看到一個典型演化：一開始用 JSON 快速做出功能，之後資料開始累積，就把它拆成正規化表。這就是很多產品真實的成長方式。

Operational Notes

幾個很容易搞混，但你一定要知道的運作細節

初學者最常被這些點卡住，因為表面上看起來是 DB 問題，實際上是「資料層角色」或「SQLite 行為」沒分清楚。

WAL 已經開啟

App 連線是透過共用的 connect_sqlite() helper 進來，會設定 journal_mode = WAL。這可以讓 SQLite 在有讀寫混合時更穩定，不容易卡成最差狀態。

foreign_keys 是 connection-scoped

這句很重要。你用 app 開的連線可以是 ON，但你如果自己用 terminal 另開一個 SQLite CLI session，看到的可能仍是 0。不是 app 沒設定，而是 SQLite 的 FK 開關是「每條連線自己決定」。

主庫和執行期狀態不是同一件事

product_state.sqlite3 是產品真相，query_runtime.sqlite3 是工作狀態。前者要穩，後者要能清。兩者角色完全不同。

最新快照和歷史資料也不是同一件事

jobs_latest.json 是「目前結果」，market_history.sqlite3 是「過去每次結果」。前者偏展示，後者偏分析。

Maintenance

現在這套 DB 系統為什麼已經夠用，以及你該怎麼維護它

你之前說你不要無限擴張，只想先做有意義的維護。這個方向是對的。下面就是目前最重要的維護原則。

原則 1：先維持分層清楚

不要因為方便，就把執行期資料、最新快照、產品主狀態全部混進同一個 DB。你現在的分層其實是健康的。

原則 2：讓會膨脹的資料有 retention

cache 要有上限，history 要有保留期限。你已經有 cache 清理與 history retention，這很重要。

原則 3：當資料要被查詢時，再考慮正規化

不是所有 JSON 都要拆。只有當它開始變胖、要被查、要被比對、會拖累維護時，才值得拆表。

原則 4：共用連線設定，不要每個模組自己亂配

你現在用統一的 SQLite helper 是正確的。這種基礎設施統一，長期非常重要。

原則 5：優先觀察真實壓力，再升級

只有真的出現多實例併發寫入、跨機器協調、超大量查詢或大型分析需求時，才考慮 PostgreSQL 或更重的 queue 系統。

原則 6：產品資料穩定，比分析資料完整更優先

收藏、通知、saved search 的一致性，通常比技能趨勢報表更重要。先保產品主功能穩，再擴分析層。

現在不需要做的事

  • 不用急著換 PostgreSQL。
  • 不用急著把所有 JSON 拆成十幾張表。
  • 不用急著做分散式 job queue。
  • 不用急著上很重的 ORM 架構重整。

真的該升級的訊號

  • 多個 app instance 同時寫主庫。
  • SQLite 鎖競爭明顯升高。
  • 你開始做大量歷史趨勢分析。
  • cache 和 history 的容量成長遠超過可控範圍。

你接下來最實用的學習方向

  • 先學看 table 和 row。
  • 再學看 primary key、unique、index。
  • 再學基本 SQL：SELECT、INSERT、UPDATE。
  • 最後再回頭看 schema 設計和 transaction。
Mini Examples

用你這個專案做幾個簡單例子

如果你會看下面這些例子，你就已經開始進入「能讀懂 DB 系統」的階段了。

例子 1：為什麼 favorite_jobs 應該在 DB

因為使用者會新增、取消、重新查詢收藏，而且這個資料跟使用者有明確關聯，也會被長期保留。這就是典型 DB 資料。 

favorite_jobs
- id
- user_id
- job_url
- created_at

例子 2：為什麼 jobs_latest.json 適合做最新快照

因為前台很多時候只想快速拿到「現在最新結果」，而不是每次都去拼完整歷史資料。這時候一份現成 snapshot 很有效率。 

jobs_latest.json
{
  "generated_at": "...",
  "queries": [...],
  "jobs": [...],
  "skills": [...],
  "task_insights": [...]
}

例子 3：為什麼 known_job_urls 後來要拆表

一開始放 JSON 很快，但資料一多就不容易查詢和維護。拆成 saved_search_seen_jobs 後，每個 URL 都是一筆 row，資料操作就正常很多。

例子 4：為什麼 market history 是另一個 DB

歷史分析資料的成長方式，和產品主庫不同。把它獨立出來，可以讓前台和產品功能不被歷史分析拖慢。

FAQ

你現在大概會問的問題

下面這些問題很像剛開始學 DB 時會卡住的點，我直接幫你回答。

Q: 一個專案只能有一個 database 嗎？

不需要。重點是資料角色清不清楚。你現在這個專案就有多個資料檔，但分工合理，所以沒問題。

Q: JSON 跟 DB 誰比較好？

沒有絕對。JSON 適合快照和輸出；DB 適合查詢、更新、關聯、一致性。你這個專案兩者都有用，而且這是合理的。

Q: 我現在要學 SQL 嗎？

要，但先學最基本的就夠。你先會看懂一張表、知道一筆 row 是什麼，再學 SELECT、WHERE、INSERT 就很夠用了。

Q: 為什麼不用一開始就上 PostgreSQL？

因為你的資料量和部署型態目前還不需要。過早升級只會增加維運成本，沒有對等收益。

Q: 現在這套 DB 會不會很脆弱？

以你目前階段來說，不會。它已經有 WAL、retention、normalized seen jobs、managed close 等保護。短期內最大的風險反而是 cache 容量，不是主庫架構。

Q: 如果我之後要加功能，要先想什麼？

先問自己三件事：這是長期資料嗎？會被查詢嗎？會跟別的資料產生關聯嗎？如果三個答案都偏是，那大概率該進 DB。

這份教學的核心結論只有一句話：你現在的 Database 系統已經不是「亂存資料」，而是「有分層、有角色、有保護機制」的系統。你短期內最重要的不是再擴充，而是真的看懂目前這套結構，之後每加一個功能，都先判斷它應該屬於快照、主庫、提交庫、runtime 還是歷史庫。
