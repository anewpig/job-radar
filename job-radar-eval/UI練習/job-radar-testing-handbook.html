Testing Handbook

Job Radar 測試手冊

這份頁面不是在講「測試理論」而已，而是直接用你現在這個 Job Radar 專案當教材，拆出前端、後端、抓取、AI、資料層、維運層到底要怎麼測， 以及你在面試裡要怎麼講自己的 testing strategy，才不會只停留在「我有寫 unit test」。

這頁要回答的問題

測什麼、為什麼這樣測、哪些要 mock、哪些一定要做整合測試、哪些只能靠手動回歸、AI 又該怎麼驗證。

這頁適合怎麼用

一方面拿來學 testing 思維，一方面也能變成你面試時的講稿底稿，說明你如何把 side project 做到可維運。

核心原則

先保護資料邊界、流程邊界、風險最高的行為，再補畫面細節。

這個專案最該測

crawl runtime、product store、query runtime、RAG assistant、connector parsing、resume analysis。

最常見誤區

只有 py_compile、只有 happy path、AI 不測 fallback、外部來源全用真網路打。

Why

1. 為什麼這個專案的 testing 特別重要

因為這不是單純 CRUD 網站。這個專案同時有 UI state、背景排程、外部站點、AI 功能、資料庫、通知、備份維運。每一層出問題，表現形式都不一樣。

它是多流程系統

使用者手動查詢、自動刷新、履歷分析、AI 問答、通知推送都是不同的流程，不可能只靠一兩個 smoke test 蓋住。

它碰外部不確定性

104、1111、Cake、LinkedIn 的頁面、節流、403/429、detail API 都會變，所以 parsing 與 retry 策略一定要被測。

它有 AI 不穩定性

LLM 的輸出天然不穩，所以必須測 answer mode、context 組裝、fallback、guardrail，而不是只驗 happy path 字串。

一句話講 testing 思維

不是為了把 coverage 數字衝高，而是為了讓「資料流、執行流、產品行為」在變動頻繁的情況下仍然可預期。

Strategy

2. 這個專案適合的測試策略

最適合的是「以風險為中心」的 testing pyramid，不是教科書式地只背 unit / integration / e2e 三個名詞。

最底層：純邏輯單元測試

skill extraction、role scoring、query signature、parsing、runtime helpers、DB helper。

→

中層：服務與資料流整合測試

crawl application service、query runtime、product store、notification service、rag assistant。

→

上層：手動回歸 / UI smoke

Streamlit 頁面、launcher、drawer、登入 dialog、複雜排版與互動。

要大量寫 automated tests 的地方

純邏輯、store、runtime、service、parsing、AI prompt builder、resume matcher。

要混合測的地方

通知、爬蟲 pipeline、background flow、snapshot sync，通常需要 mock 外部依賴但保留流程完整性。

不要只靠 automated tests 的地方

Streamlit 的真實視覺與交互、CSS 疊層、浮動按鈕、手機版排版。這些一定要手動回歸。

Frontend

4. 前端怎麼測

這個專案前端是 Streamlit，所以 testing 的重點不是 React component snapshot，而是 state、navigation、render helper 與手動回歸清單。

前端測試一

測 UI 邏輯，不先硬測 CSS

  • 先測 router 是否正確分組與切頁，例如 tests/test_ui_router.py。
  • 測 render helper 是否在資料髒掉時仍能輸出合理 fallback，例如 tests/test_ui_renderers.py。
  • 測 boot loader 這類 session flag 行為，例如 tests/test_loading_shell.py。
前端測試二

保留手動回歸清單

  • 登入 dialog、launcher、drawer、footer links、浮動按鈕。
  • 桌機 / 手機寬度。
  • hard refresh 與一般 rerun。
  • z-index、sticky、fixed 元件的疊層問題。

該 automated
router、session helper、render fallback

該 manual
版面、浮動元件、響應式

常犯錯
把 UI 邏輯與樣式耦在一起，結果很難測

改善方向
把 sections / actions / state 拆開，現在你已經在做

Backend

5. 後端與服務層怎麼測

後端最重要的是把「資料流會不會斷、狀態會不會錯、邊界條件會不會壞」測清楚。

Product Store

tests/test_product_store.py 已經示範很好的方向：user-scoped saved searches、favorites、notification preferences、AI monitoring、password reset、legacy backfill。

Query Runtime

tests/test_query_runtime.py 重點在 queue dedupe、single worker lease、partial → final snapshot、retryable failure、signal store。

Crawl Application Service

tests/test_crawl_application_service.py 是最值得拿出來講的一支，因為它把 due search、enqueue、finalize、background worker、subscriber sync 串成一條流程。

這一層最重要的觀念

測服務層時，不是只驗一個 function 回傳值，而是驗狀態轉移是否正確。像 queue 狀態、snapshot 狀態、saved search sync 結果，這些才是系統真正的契約。

Crawler

6. 爬蟲與 pipeline 怎麼測

外部來源是最不穩定的部分，所以要把 parsing、retry、fallback、cache 邏輯測得比 UI 更兇。

重點一

測 parser，不要只測真網路

  • 直接給 104 / LinkedIn / Cake 的 sample payload。
  • 驗 title、company、location、salary、detail sections 是否正確映射。
  • 這類測試又快又穩，失敗時也好定位。
重點二

測 denial 與 fallback

  • 403 / 429 要重試嗎？
  • 沒有 fresh result 時要不要吃 stale cache？
  • 沒有 cache 時要不要 soft warning？
source .venv/bin/activate python -m unittest tests.test_connectors tests.test_pipeline_staging tests.test_detail_parsing
面試說法

我知道爬蟲最脆弱的是外部來源變動，所以我把高風險的 parsing 與 denial/fallback 行為盡量固定成 deterministic tests，而不是只在真實網站上碰運氣。

AI

7. AI 不是不能測，而是要換測法

LLM 不適合用「完全一字不差」的方式測，但非常適合測：context 組裝、mode routing、fallback、guardrails、ranking、structured parsing。

適合測的

  • answer mode 分類
  • top-k 選擇
  • prompt 是否包含 guardrails
  • resume context 是否被裁剪與遮罩
  • fallback 是否在缺資料時觸發

建議半結構測

  • 市場摘要段落有沒有標題區塊
  • 比較報告有沒有 strengths / gaps
  • 引用是否有回傳

不建議硬測的

  • 完整自然語句逐字比對
  • 模型風格或修辭
  • 外部 API 真實延遲與費用行為

RAG Assistant

tests/test_rag_assistant.py 已經有 mode classification、prompt guardrails、retrieval query、comparison/guidance/market sections parsing。這很值得保留並繼續擴充。

Resume Analysis

tests/test_resume_analysis.py 示範了規則抽取、masked text、condensed context、matcher ranking、semantic floor 等很好的 AI 測試觀念。

最該補的

budget / latency 警戒值、prompt regression snapshots、evaluation dataset、模型切換後的 regression checklist。

DB

8. DB 與資料生命週期怎麼測

這個專案的資料層不是一個檔就結束，而是 product state、runtime、history、user submissions 多層邊界，所以 DB testing 要兼顧 schema、索引、資料遷移與維運。

Store correctness

主鍵、唯一性、user scope、seen jobs backfill、favorite 與 board 關聯，這些都要由 store tests 保護。

Runtime correctness

queue lease、retry 次數、partial / final snapshot 替換、signal cleanup，這些要由 runtime tests 保護。

Lifecycle correctness

backup、restore drill、retention、prune、manifest、health report，這些是維運層級測試，不做就等於沒準備上線。

source .venv/bin/activate python -m unittest \ tests.test_product_store \ tests.test_query_runtime \ tests.test_market_history_store \ tests.test_sqlite_backup_service \ tests.test_sqlite_restore_drill
Mocking

9. 什麼要 mock，什麼不要 mock

mock 用錯會讓測試變成自我安慰。原則是：外部不穩定依賴要 mock，自己的核心流程不要 mock 到看不見。

適合 mock

  • HTTP 回應、403/429、HTML payload
  • LLM client 回傳
  • 通知發送通道
  • 時間、環境變數、外部 secrets

適合 fake / temp data

  • SQLite test DB
  • snapshot 檔案
  • resume 文字樣本
  • saved search payload

不要過度 mock

  • 不要把 store 自己 mock 掉還說自己測了 service
  • 不要把 runtime queue 全 stub 掉，這樣測不到狀態轉移
  • 不要把整個 RAG flow 全部 patch 成常數字串
好測試的判準

如果外部依賴換掉，但你的核心流程邏輯不變，測試仍能抓出真問題，這就是好的 mock 邊界。

最務實的順序

先 py_compile → 再 import app → 再跑你這次碰到的功能鏈測試 → 最後做手動回歸。這比一次把所有測試全跑完更符合開發節奏。

Interview

12. 面試時怎麼講你的 testing strategy

你不要只說「我有寫單元測試」。面試官更想知道的是：你怎麼決定先測哪裡、怎麼處理外部依賴、怎麼看待 AI 測試與回歸風險。

講法一

先講風險排序

這個專案的最高風險不是單一函式，而是資料流與外部依賴，所以我優先測 runtime queue、store、crawl service、AI routing、connector parsing，而不是先追 UI pixel-perfect automation。

講法二

再講層次化方法

我把 testing 分成 pure logic、service/data flow、manual regression 三層。純邏輯多寫 automated tests；UI 視覺重手動回歸；AI 則測 context、routing、fallback，而不是逐字比對。

90 秒版本

我對 testing 的做法是風險導向，而不是 coverage 導向。像 Job Radar 這種有爬蟲、背景排程、AI、資料庫與 Streamlit UI 的系統，我會先保護 parsing、runtime queue、product store、crawl application service、RAG assistant 這些最容易讓資料流斷掉的地方。外部來源與模型回應我會 mock，但我不會把自己的核心流程整段 mock 掉。最後我會保留一份手動回歸清單，專門處理浮動元件、dialog、sticky 與手機版排版，因為這些不適合只靠自動化測試。

Roadmap

13. 如果要再把 testing 做得更完整，下一步是什麼

現在這個專案已經有不錯的測試底子，但如果你真的要把它再往產品級別推，還有幾件事值得補。

測試分組與 test matrix

把 UI、crawl、AI、DB、ops 的命令分組，讓每次改動能快速選對測試集合，而不是全跑或全不跑。

AI evaluation dataset

建立固定問題集、固定 resume 樣本、固定 snapshot 樣本，讓 prompt 或模型切換後能做 regression。

CI 與 pre-merge 規則

至少把 py_compile、import app、核心服務測試與 lint/check 放進 CI。

手動回歸腳本化

把目前的人腦清單整理成 release checklist，之後可以再逐步轉成更正式的 UI automation。

測試資料治理

為 parser、resume、AI prompt、notification payload 準備專用 fixtures，避免測試資料散落在各檔案。

測試報告化

把哪些模組有 automated tests、哪些只能手動回歸整理成文件，面試與交接都很有用。
