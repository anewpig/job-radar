Product & System Playbook

Job Radar 產品思考與系統設計手冊

這份頁面不只是講架構，而是把這個專案當成一個真的要走向產品化的案例，從使用者價值、功能切分、資料與狀態設計、 同步與非同步流程、AI 位置、資料庫分層、營運指標、擴充路線，一路講到你未來要怎麼做決策。 你如果之後要自己做一個產品，真正要學的不是「某個元件怎麼寫」，而是「為什麼這個系統要這樣切」。

你真正該看什麼

  • 這個產品的核心價值不是爬資料，而是把求職決策變成持續更新的工作台。
  • 功能應該怎麼從使用者問題往下切，不是從技術模組往上湊。
  • 哪些流程要同步，哪些流程要背景執行，為什麼。
  • 狀態與資料的 source of truth 應該放在哪一層。
  • AI 應該是產品的哪一層能力，不該直接變成唯一核心。
  • 如果要擴充成多人、多來源、企業版、可營運的產品，要怎麼演進。

01 Product Thesis

先定義產品，不要先定義技術

這個專案表面上像「職缺爬蟲」，但如果你把它只當成爬蟲，你會把產品做小。 它真正能成為產品的地方，是把「市場變化、個人背景、履歷匹配、投遞節奏、通知、看板、AI 助理」整合成一套求職工作台。

核心承諾

我幫你持續整理你所在職位市場，讓你知道哪些技能值得補、哪些職缺值得投、哪些變化值得追。

產品單位價值

不是一筆 job listing，而是一個「可持續刷新、可被比較、可被解釋」的市場快照與決策工作台。

最容易走歪的地方

如果只追求抓得更多、畫面更花、AI 回答更長，卻沒有讓使用者更快做決策，產品會很快失焦。

這個專案若要產品化，定位應該怎麼講
層次 錯的講法 更對的講法 為什麼
技術層 這是一個職缺爬蟲。 這是一個市場資料採集與分析引擎。 爬只是手段，不是產品價值本身。
功能層 它可以抓 104 / 1111 / Cake / LinkedIn。 它可以把多平台職缺彙整成同一份市場快照與行動清單。 使用者在意的是決策整合，不是來源數量本身。
產品層 它幫你找工作。 它幫你持續理解市場，並把求職行動系統化。 找工作不是一次性搜尋，而是持續迭代的過程。
商業層 它是個工具。 它是求職 operating system 的雛型。 這樣你未來才有機會往訂閱、企業協助、教練化服務擴。
02 Users & JTBD

先想使用者在解什麼問題

產品設計不是先列功能，而是先搞清楚誰痛、什麼時候痛、哪一個決策點最卡。 這個專案最適合的切法是用 JTBD，而不是只按使用者身份分類。

JTBD 1：市場摸底

「我想知道 AI 工程師 / 應用工程師 / PM 目前市場到底在要什麼，不想一個平台一個平台慢慢查。」

對應功能：搜尋工作台 市場快照 技能 / 工作內容 insight

JTBD 2：履歷校正

「我想知道我的履歷跟這批市場缺口在哪，先補什麼最有用。」

對應功能：履歷分析 AI 匹配 缺口建議

JTBD 3：持續追市場

「我不要每次重頭搜尋，我想讓系統自動幫我追某些條件，有新結果再通知我。」

對應功能：saved search scheduler / worker 通知

JTBD 4：整理行動

「我不只是看職缺，我還要追蹤收藏、投遞進度、下一步。」

對應功能：tracking board favorites

JTBD 5：快速理解資料

「我不想自己解讀一堆卡片與圖表，我想直接問：目前要補什麼？哪個方向值得投？」

對應功能：AI 助理 RAG launcher

JTBD 6：建立長期節奏

「我希望這不是一次性工具，而是一個會陪我持續迭代的求職系統。」

對應功能：通知 歷史資料 個人設定

04 Product Loop

這個產品真正的成長迴圈

一個工具能不能變成產品，取決於它有沒有 loop。這個專案的 loop 不是社交網路那種 viral loop， 而是「資料迴圈 + 行為迴圈 + AI 解讀迴圈」。

資料迴圈

使用者定義搜尋條件 → 系統抓資料 → 形成市場快照 → 保存為 saved search → 定時刷新 → 新資料再回到使用者。

決策迴圈

使用者看市場 → 履歷分析 → 發現缺口 → 調整履歷 / 投遞方向 → 再回來看新的市場結果。

AI 解讀迴圈

使用者提問 → RAG 助理解釋 → 使用者縮小範圍或問更深入問題 → 更快收斂到具體行動。

如果你要做產品，loop 比功能清單更重要
迴圈 觸發物 回饋物 產品價值 沒有這個迴圈會怎樣
資料迴圈 saved search / scheduler 新職缺、快照更新、通知 讓產品具有持續性 產品會退化成一次性搜尋工具
決策迴圈 resume analysis / tracking / board 補強方向、投遞動作 把資料轉成行動 使用者看完資料卻不知道下一步
AI 解讀迴圈 assistant / launcher 摘要、比較、個人化建議 降低理解成本 功能很多，但認知門檻高
05 Feature Stack

功能不是一坨，應該按價值鏈拆

你如果之後要再加功能，不要按頁面名稱想，而是按價值鏈想：發現、理解、決策、行動、保留。

發現層

  • 搜尋設定
  • 多平台抓取
  • overview / market overview
  • 來源分布 / 匹配角色

理解層

  • 市場技能 insight
  • 工作內容 insight
  • AI 助理
  • 履歷匹配與缺口分析

行動層

  • 收藏
  • 投遞看板
  • saved search
  • 通知設定

保留層

  • 快照歷史
  • 自動刷新
  • 通知
  • 個人 profile / 履歷保存

這樣拆的好處

你會知道新功能到底是在補哪一層價值，而不是只覺得「這頁還能再加一個按鈕」。

常見錯誤

只圍繞某個頁面加功能，最後頁面越做越胖，但整體使用者旅程沒有變更完整。

06 Product Thinking

如果你要做一個產品，應該怎麼思考

這段是最核心的。你以後做任何產品，都應該先回答下面這些問題，再開始寫系統。

問題 1：產品的最小承諾是什麼

對這個專案來說，不是「我會抓很多資料」，而是「我會讓你更快知道市場與自己的落差」。

問題 2：使用者第一次何時覺得有價值

當他輸入角色後，能在短時間看到第一批有解釋力的市場快照，而不是只看到 loading。

問題 3：第二次為什麼會回來

因為系統會記住他的搜尋、更新快照、有新職缺通知，並累積個人背景與歷史資料。

問題 4：什麼是 source of truth

市場快照、saved search、履歷 profile、board 狀態、AI 監控，都應該有明確存放層，而不是都塞 session。

問題 5：哪裡需要同步，哪裡需要背景化

第一次價值需要同步快速回應；定期刷新與通知應該背景化；AI 長任務之後也可能要 background 化。

問題 6：什麼狀況算失敗

不是只有 exception。資料太舊、沒有個人背景、通知綁定失效、搜索條件不合理，也都算產品層失敗。

你做產品時應該先回答的問題
問題 在這個專案裡的答案 如果答不出來會有什麼後果
使用者來這裡是為了什麼 理解市場、校正履歷、追蹤新機會、整理求職行動。 功能會長很多，但方向混亂。
你的 unit of value 是什麼 一份市場快照 + 一組持續追蹤的搜尋 + 一套個人化決策輔助。 你會錯把某個局部功能當成整個產品。
第一個 wow moment 是什麼 快速看到多平台整合後的市場結果與 insight。 使用者還沒看到價值就先離開。
留存機制是什麼 saved search、自動刷新、通知、看板、歷史資料。 產品只有新鮮感，沒有回訪理由。
AI 在哪裡最有價值 解讀市場、解釋缺口、摘要複雜資訊，而不是取代核心資料流程。 你會把 AI 放在錯誤位置，成本高又不穩。
07 System Principles

系統設計的核心原則

你如果想把這個專案做成產品，系統設計就不能只看 class 與 function，而是要先有幾條穩定原則。

原則一：先有產品節奏，再有技術拆分

同步與背景流程怎麼切，要看使用者等待容忍度，不是看你喜歡哪種架構。

原則二：每一份狀態都要有主人

session、快照、saved search、board、notification preference、AI event，不能模糊混放。

原則三：AI 不是 source of truth

AI 應該負責解讀、排序、補強，不應取代資料採集與核心狀態管理。

原則四：每條長流程都要可恢復

爬蟲、AI、通知、背景刷新，都要能知道做到哪裡、失敗在哪裡、怎麼續跑。

09 State Ownership

狀態設計：每一份資料該放哪裡

這是產品系統最重要的題目之一。很多系統後來難維護，不是因為技術不夠，而是因為 state 放錯地方。

這個專案的 state ownership 地圖
狀態 應該放哪裡 為什麼 如果放錯地方會怎樣
當前 UI tab、draft text、臨時 pending state st.session_state 它是互動中的暫存狀態，不值得長久保存。 若存到 DB，會讓持久層混入一堆垃圾互動狀態。
saved searches、favorites、board、notification preferences、auth product_state.sqlite3 / ProductStore 這些是產品主資料，應該長期保存並與 user 綁定。 若只放 session，重新整理就沒了；若拆散在很多檔案，資料一致性會很差。
crawl jobs、lease、runtime snapshots query_runtime.sqlite3 + snapshot dir 這些是執行狀態，不該跟產品主資料混在一起。 會讓產品 DB 又大又亂，也難做 scheduler / worker 維護。
市場歷史 archive market_history.sqlite3 歷史資料量大、用途不同，應與即時產品狀態分層。 放進主 DB 會拖慢查詢，也會讓 schema 難演化。
匿名化 profile / 補充提交 user_submissions.sqlite3 這塊屬於收集型資料，不該直接混到產品主 state。 資料治理與權限邊界會模糊。

現在這樣分層的優點

  • 產品資料、執行資料、歷史資料、收集資料各自有邊界。
  • 做備份、維護、健康檢查時比較容易針對性處理。
  • 未來如果要拆服務，也比較知道每塊資料應該跟誰走。

未來要注意的點

  • 跨 DB 的一致性要靠應用層協調，而不是幻想有交易。
  • query signature、search_id、user_id 要維持穩定，不然會很難對資料。
  • 如果之後多機部署，檔案 snapshot 與 cache 要重新考慮共享策略。
10 Sync vs Async

哪些流程應該同步，哪些應該非同步

很多人做產品時會把所有事情都做成同步，因為簡單；或反過來把什麼都 background 化，因為看起來很專業。 其實正確答案只有一個：看使用者等待時是否還能感受到價值。

適合同步的

  • 第一次搜尋後的初始波結果
  • 履歷分析的第一輪回饋
  • AI 助理的一般提問

因為這些是「第一次價值」的一部分，等待可以接受，但一定要有明確 staged feedback。

適合背景化的

  • saved search 定時刷新
  • 通知發送
  • 備份、維護、健康檢查
  • 未來的大型 AI 報告

因為使用者不必卡在畫面上等，而且這些工作可以排隊、重試、監控。

這個專案目前的平衡點

  • 搜尋有 staged crawl 與 queue。
  • saved search 有 scheduler / worker。
  • AI 仍以同步互動為主，因為產品還在強化互動體驗期。
同步 / 非同步設計判準
流程 現在做法 為什麼 未來何時應該改
初次查詢 前台等待第一波結果，再逐步補 detail / finalize。 使用者需要立即看到價值。 若查詢量暴增，可改成先讀舊 snapshot + 背景 refresh。
履歷分析 同步執行，畫面顯示狀態。 這是高意圖、低頻但高價值動作，等待可接受。 若分析流程拉長或加 OCR / 多模型，就應 background 化。
AI 助理提問 同步執行。 對話互動若轉背景，體驗會明顯變差。 只有在要做超長報告或多步 agent 才考慮切背景。
saved search 刷新 scheduler / worker 背景處理。 天然非同步，也需要重試與 lease。 量變大時可從 SQLite queue 升級到更專門的 queue。
11 Design Decisions

系統設計關鍵取捨：這個專案為什麼這樣做

這裡不是在列「最佳實踐」，而是直接討論這個專案現在的選擇為什麼合理，還有什麼時候會不再合理。

單體應用 + 模組化分層

為什麼：產品還在快速迭代，需要低摩擦調整 UI、服務、資料層。

優點：開發快、跨層改動容易、部署簡單。

缺點：當併發與團隊變大時，邊界治理會變難。

替代：前後端分離、微服務、BFF。

SQLite 多庫分層

為什麼：單機產品最划算，且不同資料型態天然就適合不同庫。

優點：簡單、便宜、可攜、好備份。

缺點：跨庫一致性與多機共享能力有限。

替代：Postgres + dedicated queue + object storage。

Query Signature 作為查詢身份

為什麼：同條件查詢應該共用快照與背景刷新邏輯。

優點：好 dedupe、好 cache、好做 scheduler。

缺點：簽名邏輯若變，舊資料遷移要小心。

替代：顯式 SearchSpec table + normalized query graph。

AI 疊在資料層之上

為什麼：資料採集與產品狀態是核心，AI 是解讀與增強層。

優點：即使 AI 不可用，產品仍可運作。

缺點：會有兩套邏輯：AI path 與 non-AI path。

替代：AI-native first product，但穩定性與成本風險更高。

前台 staged feedback

為什麼：使用者能感覺到系統正在做什麼，比單一 spinner 更有信心。

優點：等待感受更好，也更便於 debug。

缺點：流程邏輯與 UI 狀態會比較複雜。

替代：背景任務 + notification，或更簡化的 skeleton-only 策略。

保存歷史與當前分開

為什麼：當前快照重可用性，歷史資料重分析深度。

優點：兩種查詢模式不互相污染。

缺點：要多維護一套 history store。

替代：單一分析倉儲，但產品即時體驗通常會變差。

12 Metrics

如果這是產品，你要看哪些指標

很多人把技術監控和產品指標混在一起。實際上你需要三層：產品使用、系統健康、AI 質量 / 成本。

產品使用指標

  • 首次搜尋完成率
  • 建立 saved search 比率
  • 履歷分析使用率
  • AI 助理提問率
  • 通知綁定率
  • 回訪率 / 留存率

系統健康指標

  • crawl success rate
  • snapshot freshness
  • queue backlog
  • worker heartbeat
  • notification send success
  • DB health / backup status

AI 指標

  • resume build success rate
  • assistant answer success rate
  • blocked_missing_profile 比例
  • AI latency
  • token per request
  • citation usage / used_chunks
建議你真的要看的 KPI
類型 KPI 這個專案為什麼重要 如果變差代表什麼
Activation 首次看到市場快照的比例 這是產品第一個價值點。 搜尋設定或爬取流程太慢 / 太複雜。
Retention saved search 建立率與回訪率 如果沒有這個，產品會變成一次性工具。 沒有持續價值、沒有回來理由。
Action 收藏 / 看板使用率 這表示產品是否從「看資料」跨到「做行動」。 資料很多，但沒有形成 workflow。
Quality 履歷分析成功率與 AI 問答成功率 這兩條是高價值功能，一旦不穩，信任會下降很快。 模型、資料品質、流程穩定性有問題。
Cost 每次問答 / 每次履歷分析 token 成本 AI 成本一高，產品規模就會被卡死。 cache 不夠、模型太大、prompt 太肥。
13 Mistakes

如果你繼續做這個產品，最常見會踩的坑

這一段我刻意寫得直接一點。因為產品做到這一步，最大的風險不是寫不出功能，而是方向做錯。

坑 1：把抓更多來源當成唯一成長

來源當然重要，但如果理解層、通知層、行動層沒跟上，只會讓資料更多、更亂。

坑 2：過度迷信 AI

AI 應該幫你解釋與排序，不是取代資料採集、狀態管理、產品邏輯。

坑 3：所有東西都做成同步

會讓等待時間變長、互動變卡，也很難做 scheduler / worker 維運。

坑 4：所有資料都塞一個 DB

最後維護、備份、健康檢查、遷移都會痛。

坑 5：UI 頁面越做越胖

如果功能切分只跟頁面走，很快就會失去產品邏輯與模組邊界。

坑 6：沒有定義留存機制

使用者看一次市場很新鮮，但沒有 saved search / 通知 / 歷史資料，第二次就不會回來。

14 Roadmap

如果要把這個專案做成真正的產品，建議路線圖

不是一次把架構全換掉，而是按產品成熟度逐段升級。以下這四階段，是我覺得最合理的演進路徑。

Stage 1：單機產品化

把現在這套 UI、queue、AI、DB 手冊化、監控化、回歸化。這是把 prototype 變成可穩定營運的版本。

Stage 2：流程產品化

強化 saved search、通知、看板、歷史資料與 KPI，讓留存機制完整。

Stage 3：服務拆分

當負載上來後，再把 crawl、AI、通知、前端介面逐步切開，不要提早微服務化。

Stage 4：企業 / 多租戶

加治理、RBAC、審計、共享隊列、外部 cache、Postgres / object storage、SLA 與告警。

15 Expansion

未來要擴充時，你應該怎麼切

下面不是抽象方向，而是針對這個專案，當你要擴新功能時，最應該下刀的位置。

新增來源平台

去 connector 層與 pipeline 層擴，不要碰 UI 主邏輯。重點是統一成相同的 JobListing 與 detail parsing 結構。

新增 AI 能力

去 resume/* 或 assistant/* 擴，不要讓 AI 反過來綁死抓取流程。先問清楚 AI 是補「理解層」還是「行動層」。

做團隊版 / 顧問版

加 user / workspace / role / shared board / shared searches，不要先去改畫面，先想資料模型與權限模型。

做營收

最自然的是從高價值流程切：高頻刷新、深度 AI 報告、履歷教練、求職記錄與通知、自訂監控、顧問協作。

擴充時的建議切點
想擴什麼 先切哪一層 這樣做的優點 不要先做什麼
更多來源 connectors / pipeline / analysis mapping 既有產品層不必大改 不要直接在頁面上加平台特例
更強 AI 問答 assistant service / retrieval / chunk schema 維持產品邊界清楚 不要先把所有資料都灌進大 prompt
更多用戶與併發 runtime queue / store / cache / deployment 先穩住系統壓力點 不要只改前端 loading 樣式
企業版 auth、RBAC、audit、workspace 資料模型 避免後面權限模型返工 不要先只加一個「公司模式」開關
16 Checklist

如果你要繼續把這個專案做成產品，下一輪最值得做的清單

我把這份清單分成產品、系統、AI、營運四類。你不需要一次全做，但至少要有這份地圖。

產品面

  • 把首次體驗再收斂，縮短從輸入到第一批結果的時間。
  • 把 saved search 建立率做成明確引導與 KPI。
  • 讓 tracking / board 更像工作流，而不是附加頁面。
  • 把通知與回訪理由做得更清楚。

系統面

  • 做完整手動回歸腳本與 smoke checklist。
  • 補 queue、snapshot freshness、notification、DB 健康指標。
  • 把備份 / 還原 / drill 真的納入營運節奏。
  • 規劃多機與共享 cache 的升級路線，但先不要急著改。

AI 面

  • 建立固定的問題集與履歷樣本做離線回歸。
  • 把 AI 監控拆到 retrieval / generation / citation 更細粒度。
  • 為助理補一層更穩的 fallback answer 策略。
  • 整理 prompt / schema / model policy 文件。

營運面

  • 定義核心 KPI 與每週檢視儀表板。
  • 建立失敗類型分類：資料、AI、通知、使用者設定。
  • 規劃資料保留與清理策略。
  • 把產品說法從「爬蟲」轉成「求職工作台」。
一句話總結這份手冊

這個專案真正該走的方向，不是把每個頁面做得更滿，而是把它明確做成一個「會持續更新市場、會理解使用者、會幫助做決策」的求職 operating system。 產品思考的核心不是多做功能，而是讓資料、AI、流程與留存機制對齊同一個使用者價值。

這份頁面從產品思維切入，整理價值鏈、使用者回訪機制、功能優先序與系統設計判斷。
