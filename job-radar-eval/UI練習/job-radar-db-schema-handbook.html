Job Radar DB Handbook

Job Radar 資料庫與 Schema 深度手冊

這頁專門講資料層，不重複整個系統架構，而是直接回答幾個更實際的問題：目前有哪些資料庫、每份資料庫各自扛什麼責任、 每張核心表大概在存什麼、為什麼要這樣分層、哪些是執行期資料、哪些是產品主資料、哪些帶敏感資訊、哪些適合日後搬到更正式的儲存層。

一句話總結

目前資料層不是「一個 SQLite 打天下」，而是刻意拆成四種責任：產品主資料、抓取執行期 runtime、使用者提交/履歷資料、市場歷史 archive。 另外再加上 JSON snapshot / HTTP cache 承接 payload 本體。這種切法的最大價值，是未來每一層都可以獨立搬家，而不是一次性大重構。

  • product_state.sqlite3：產品主資料、通知、收藏、saved search、AI monitoring。
  • query_runtime.sqlite3：query snapshots、crawl queue、runtime heartbeat。
  • user_submissions.sqlite3：履歷與使用者提交內容的遮罩後留存。
  • market_history.sqlite3：crawl run archive 與歷史職缺。
  • JSON / Cache：完整快照 payload 與來源抓取快取。

這份頁面特別補什麼

不是只有表名，而是把責任、關聯、索引、讀寫模式、生命週期與升級時機一起講清楚。

Overview

目前資料策略的核心判斷

這套系統最值得肯定的資料層決策，不是用了哪個資料庫，而是很早就知道不同資料不該混在一起。 一般產品在早期常見的錯誤是，把帳號、背景 queue、AI 監控、履歷原文、歷史 archive 全塞同一份 DB。你的系統沒有這樣做，所以後續升級時可以一層一層搬，不用整台機器一起拆。

主資料product_state保存會直接影響產品體驗與使用者狀態的資料。
執行期query_runtime只關心 queue、snapshot registry 與系統心跳。
敏感提交user_submissions把履歷等較敏感內容與一般產品資料分開。
歷史分析market_history保存長期市場資料，不污染 live product flows。
Strategy

為什麼資料要這樣拆

資料拆分的關鍵不是技術偏好，而是生命週期、寫入頻率、敏感程度、查詢模式、未來搬遷方向都不同。

產品主資料不應與 queue 混在一起

因為產品主資料追求一致性與長期保存；queue 追求高頻短生命週期與執行期可觀察性。

S1

如果把 saved search、收藏與 crawl_jobs 放在一起，你很快就會遇到 retention、索引、鎖競爭與備份邊界都互相衝突的問題。

敏感資料不應與一般產品資料混在一起

履歷內容的治理方式與一般產品資料不同。

S2

只要一開始就分檔，未來要加 retention、刪除流程、加密、匯出與審計都會容易很多。

歷史資料不應拖慢 live 查詢

長期 archive 的成長模式，和工作台當前互動需要的低延遲讀取完全不同。

S3

歷史市場資料適合慢慢累積並拿來做趨勢；工作台則需要「現在這次查詢的結果」。把兩者分開，是正確的產品數據架構思路。

Payload 本體與 metadata 不該混在一起

完整快照內容和 registry/queue metadata 的最佳儲存方式不同。

S4

因此現在用 DB 存 metadata，用 JSON / cache 存 payload，本質上是對的。只是未來量大後要把檔案層升級成 object storage。

Storage Map

資料層總覽圖

從資料流角度看，UI 和 service 寫進的其實不是同一種資料，而是分流到五個儲存槽位。

使用者操作 → `product_state.sqlite3`：帳號、saved search、favorites、通知偏好、AI monitoring、履歷 profile。
抓取執行期 → `query_runtime.sqlite3`：query snapshot registry、crawl jobs、runtime signals。
履歷原文 / 提交內容 → `user_submissions.sqlite3`：遮罩後提交記錄與分析上下文。
每輪市場快照 archive → `market_history.sqlite3`：crawl runs、job posts、crawl_run_jobs。
完整 payload / cache → JSON snapshots 與 cache 目錄：快照本體與抓取快取。
Product DB

product_state.sqlite3

這份資料庫是產品主資料層。它承載的是使用者在工作台上的狀態與行為痕跡，而不是背景 queue 或大體積 payload。

AUTH

帳號與身份關聯群組

`users`、`user_identities`、`password_reset_tokens` 是登入與身份映射核心。

+

users

主帳號表。保存 email、display_name、password_salt、password_hash、is_guest、created_at、updated_at、last_login_at。

  • 主鍵：`id`
  • 唯一鍵：`email`
  • 特例：會自動建立 guest user

user_identities

第三方身份映射表。用 `(provider, subject)` 作主鍵，把 OAuth/OIDC provider 身份連回本地 user。

  • 主鍵：`(provider, subject)`
  • 索引：`idx_user_identities_user_provider`, `idx_user_identities_email`

password_reset_tokens

密碼重設流程的臨時令牌 / reset code。保存使用者、建立時間、過期時間、消耗時間。

為什麼這樣分

  • `users` 只負責本地帳號主體，不把第三方 provider 細節塞進去。
  • `user_identities` 讓多 provider 綁定變得自然，也避免後續 provider 擴充時破壞 `users`。
  • `password_reset_tokens` 獨立出來，讓重設流程有清楚的過期與消耗語義。

未來風險與建議

  • 若 provider 增多，要補更明確的 unique / lifecycle policy。
  • password reset token 需要 retention / cleanup，避免歷史垃圾累積。
  • 若要企業化，需再加權限模型與 audit trail。
SEARCH

搜尋與追蹤群組

`saved_searches`、`saved_search_seen_jobs`、`favorite_jobs` 是工作台長期追蹤核心。

+

saved_searches

保存使用者查詢條件，包含 rows_json、custom_queries_text、crawl_preset_label、signature、last_run_at、last_job_count 等。

  • 主鍵：`id`
  • 唯一鍵：`(user_id, name)`
  • 索引：`idx_saved_searches_user_signature`, `idx_saved_searches_user_updated_at`

saved_search_seen_jobs

記錄某個 saved search 已經看過哪些 job_url，以及第一次 / 最後一次看到時間。

  • 主鍵：`(search_id, job_url)`
  • 外鍵：`search_id -> saved_searches.id`
  • 索引：`idx_saved_search_seen_jobs_search_ordinal`

favorite_jobs

使用者收藏的職缺，同時保存投遞狀態、投遞日期、面試日期、面試筆記。

  • 主鍵：`id`
  • 唯一鍵：`(user_id, job_url)`
  • 索引：`idx_favorite_jobs_user_saved_search`, `idx_favorite_jobs_user_application_status`

設計動機

  • `saved_searches` 解的是「條件保存」；`saved_search_seen_jobs` 解的是「去重與新舊判斷」。這兩件事不該混成一欄大 JSON。
  • `favorite_jobs` 不只是收藏表，實際上已經兼具簡化版投遞看板資料來源。
  • 這個群組代表產品的黏著度來源，也是通知與自動刷新最常回寫的區域。

常見讀寫模式

  • 搜尋完成後更新 `saved_searches` 的統計與 last_run_at。
  • 自動刷新後對照 `saved_search_seen_jobs` 判斷哪些是新職缺。
  • 收藏與投遞狀態編輯走 `favorite_jobs`。

未來擴充

  • 若投遞看板變複雜，`favorite_jobs` 可能要拆成 application pipeline tables。
  • 若 saved search 很多，可考慮把 rows_json 正規化或至少補 migration/version。
NOTI

通知與偏好群組

`job_notifications`、`notification_preferences` 承接站內與站外提醒能力。

+

job_notifications

記錄某次 saved search 產生的新職缺通知，包含 new_jobs_json、已讀狀態、Email / LINE 是否送出與 delivery_notes。

  • 主鍵：`id`
  • 索引：`idx_job_notifications_user_read_created`

notification_preferences

每位使用者的通知偏好：site / email / line 開關、收件者、LINE 綁定資訊、relevance threshold、frequency。

  • 主鍵：`user_id`
  • 一使用者一筆偏好

為什麼不是只存發送結果

  • 通知本身是一個產品事件，不只是傳輸行為，所以需要留下站內記錄。
  • 偏好設定與發送記錄必須分開，否則很難重送、分析或回顧。

未來建議

  • 若通道增多，將 provider 細節抽成 provider registry 或 delivery attempt table。
  • 若通知可靠性很重要，發送結果應獨立 attempt log，而不只是一組布林欄位。
META

產品運營與 AI 監控群組

`app_metrics`、`ai_monitoring_events`、`user_resume_profiles` 提供輕量營運與個人化基底。

+

app_metrics

目前是簡單 key-value metrics 表，例如 total_visits。是最輕量的營運數值承載點。

ai_monitoring_events

保存 AI 相關事件，包含 event_type、status、latency_ms、model_name、query_signature、metadata_json、created_at。

  • 索引：按 event_type / user_id / status + created_at
  • 用途：AI 成本、延遲、失敗率與回退策略觀察

user_resume_profiles

保存使用者目前最新的結構化履歷 profile，讓履歷頁與 AI 助理能重用個人化輪廓。

為什麼放在 product_state

  • 這些資料會直接影響產品體驗或個人化，而不是單純背景維運。
  • `user_resume_profiles` 雖然和履歷有關，但它是衍生後的產品可用 profile，不等於原始提交內容。

何時該搬

  • `ai_monitoring_events` 變大時，應搬去 analytics / observability store。
  • `app_metrics` 如果從 key-value 演變成正式儀表板，就該進 metrics pipeline。
Runtime DB

query_runtime.sqlite3

這份 DB 不承載產品主資料，而是承載抓取執行期 metadata。它的核心價值不是資料豐富，而是讓 queue、snapshot freshness 與 heartbeat 都有可觀察狀態。

query_snapshots

以 `query_signature` 為主鍵的 snapshot registry。

QS

欄位包含：`status`, `generated_at`, `fresh_until`, `storage_key`, `last_accessed_at`, `updated_at`, `error_message`, `is_partial`。它不直接保存完整 payload，而是保存快照 metadata 與檔案 key。

主鍵：query_signature角色：registry搭配：JSON snapshots

crawl_jobs

抓取 queue。保存 priority、status、lease_owner、lease_expires_at、attempt_count、snapshot_ref 等。

CJ

這張表的設計目標是「足夠支援 dedupe、lease、retry、priority」，而不是做通用訊息佇列。它目前很務實，但也清楚知道自己不是終局 queue。

索引：signature_status索引：pending_order角色：queue

runtime_signals

scheduler / worker / cleanup 等背景元件的 heartbeat 表。

RS

主鍵是 `(component_kind, component_id)`，搭配 `updated_at` 索引。它不是完整事件流，而是最近狀態的 upsert 式快照。

主鍵：component_kind + component_id索引：updated_at DESC

這層資料最重要的特性

短生命週期、寫入頻繁、可觀察性優先。

WHY

runtime 資料本質上不是為了永久保存，而是為了讓背景流程能協作並被觀察。因此它的 retention 與升級方向，應和產品主資料完全不同。

runtime 層的典型狀態變化
表典型寫入典型讀取為什麼重要
query_snapshots生成 partial / ready snapshot 時 upsert查詢是否命中 cache、是否仍 fresh讓同一個 query 不必每次都重抓
crawl_jobsenqueue、lease、retry、complete、failworker 挑下一個 job；營運看 backlog自動刷新與背景抓取的執行核心
runtime_signalsscheduler / worker 定期 upsert heartbeat後端控制台查看是否還活著沒有它就不知道背景組件是否靜默失效
User Submission DB

user_submissions.sqlite3

這份 DB 現在非常單純，只有 `user_submissions` 一張主表，但它的重要性很高，因為它承接的是較高敏感度的使用者輸入內容。

user_submissions

保存建立時間、來源型別、遮罩後原文、summary、target_roles、core_skills、tool_skills、preferred_tasks、generated_prompts、extraction_method、llm_model、notes。

SUB

它的角色不是產品主狀態，而是留下一份「履歷/提交內容的開發與分析可用影子」。因此它應保留最小必要資訊，並更重視遮罩、刪除與 retention，而不是長期持有大量原文。

為什麼獨立成一份 DB

因為這層資料的敏感度與治理要求明顯不同。

PII

只要這些資料和 product_state 混在一起，未來刪除流程、權限隔離、備份策略都會一起複雜化。現在先分檔，是非常合理的前置治理。

History DB

market_history.sqlite3

這層代表的不是「現在這次查詢」，而是「過去所有查詢累積下來的市場歷史」。它讓系統未來有能力做趨勢、角色變化、技能變化與來源比較。

crawl_runs

每一次 crawl 的總結記錄，保存 generated_at、query_fingerprint、queries_json、role_targets_json、job/skill/task/error 計數與摘要 JSON。

RUN

這張表的角色是「批次層級的歷史檔案」，不是單筆職缺表。

job_posts

以 `job_url` 為主鍵的職缺主表，記錄 first_seen / last_seen 與 latest_payload_json。

POST

它相當於歷史維度中的職缺主實體表，讓同一個職缺可跨多個 crawl_run 被追蹤。

crawl_run_jobs

把某次 crawl_run 與當次抓到的 job_url 關聯起來，並保存當時快照、排序與 relevance_score。

MAP

它扮演 run 與 job 之間的橋接表，讓「某次跑到什麼」與「某個 job 在不同 run 如何變化」都能回答。

這層資料為什麼值錢

因為沒有 archive，你就永遠只能回答「現在」，不能回答「變化」。

WHY

歷史層真正的價值不是備份，而是讓市場分析從即時檢視升級成時間序列洞察。當你要談技能熱度趨勢、JD 變化、角色漂移，這層才是關鍵。

history 層關聯視角
表主鍵關聯查詢價值
crawl_runsid被 crawl_run_jobs 參照看某次整體輸出、摘要、錯誤與 query fingerprint
job_postsjob_url被 crawl_run_jobs 參照看某職缺第一次/最後一次出現與最新 payload
crawl_run_jobs(crawl_run_id, job_url)橋接 crawl_runs 與 job_posts看某次 run 含哪些職缺、排序與當時快照
Snapshot Layer

JSON Snapshot 與 Cache 層

除了四份 SQLite，還有一個常被低估但非常重要的層：JSON snapshot 與抓取快取。這層不是備而不用，而是當前系統可觀察性與 payload 保存策略的重要部分。

jobs_latest.json

最新一份市場快照的 materialized view。

LST

適合 UI cold start 與最近結果展示，但不應被當成長期唯一資料來源。

snapshot store

按 query signature digest 生成的 `.json` 或 `.partial.json` 檔。

SNP

搭配 `query_snapshots.storage_key` 使用，DB 存 registry，檔案存完整 payload。

HTTP / HTML Cache

來源抓取快取，提升重跑效率與 debug 能力。

HT

這層尤其適合 parser debug、比對來源改版與降低重複請求成本。

未來替代方案

當檔案數量與多機需求上升時，這層應最先遷移到 object storage。

OBJ

DB 仍留 metadata；payload 與 cache 搬到更適合共享與版本治理的儲存層。

Lifecycle

資料生命週期與搬遷時機

真正成熟的資料層，不只知道「怎麼存」，還知道「要留多久、什麼時候該搬」。

生命週期建議
資料群建議保留期間為什麼何時該搬或拆
產品主資料長期屬於產品核心狀態當多使用者寫入與 migration 需求增加時搬到 Postgres
runtime queue / signals短到中期多數只對最近執行期有價值當 worker 數量、queue throughput、跨機需求上升時換 broker
AI monitoring中期適合做近階段監控與成本分析當事件量大時拆到 analytics / observability store
履歷遮罩資料盡量短到中期敏感度高，不宜無限期保存需要刪除流程、TTL 或加密策略時
歷史市場 archive長期要支持趨勢分析歷史量變大時轉倉儲 / DuckDB / Parquet
JSON snapshot / cache視 freshness 與 debug 需求可用性與 debug 價值高，但不宜無限制堆積檔案數與多機需求升高時改 object storage + manifest

資料層下一步最合理的升級順序

如果未來要升級，不要同時重構所有資料層。比較合理的順序如下。

1. 先搬 product_state

因為它最接近正式產品主資料。

R1

一旦使用者數量與 migration 需求提升，Postgres 會比 SQLite 更適合這一層。

2. 再換 runtime queue

因為 scheduler / worker 壓力會先打到這裡。

R2

把 `crawl_jobs` 與 lease/retry 能力移到 Redis / Postgres queue，是最自然的第二步。

3. 最後整理 archive / analytics

歷史與 AI 監控層再往分析型儲存走。

R3

當要做趨勢、BI 或 AI quality dashboard 時，再把 history 與 monitoring 拆到更適合的分析層。
