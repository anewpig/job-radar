1 誰會每天打開這個產品
2 他最在意哪個畫面延遲

第一課

先定義產品與 MVP

系統設計不是從資料庫開始，而是先收斂產品的「核心價值鏈」。盯盤 Web 的價值，不是做一個大而全的金融入口，而是讓特定用戶在需要的時候看到關鍵訊號並快速反應。

先問清楚這 4 題

  • 目標使用者是短線交易者、波段投資者，還是一般投資人？
  • 核心價值是即時報價、條件提醒、新聞整合，還是策略訊號？
  • 使用者一天會打開幾次，通常在什麼時間點打開？
  • 產品成功指標是日活、提醒送達率、留存，還是付費轉換？

如果是我，第一版只做這些

  • 登入與基本帳號管理
  • 自選股清單
  • 即時或準即時報價
  • 價格 / 漲跌幅提醒
  • Web Push 或 Email 通知
  • 最基本的走勢圖與行情摘要
這一課的結論

先設計一條完整價值鏈：使用者登入、加入自選股、看到行情、設定提醒、收到通知。這條鏈要先跑通，其他都晚一點。

產品範圍

第一版不要做下單、社群、回測、多券商串接。那些都會讓系統邊界暴增，並模糊你最初的產品驗證。

你現在就該寫下來

一句產品定位、三個主要使用情境、五個 MVP 功能、五個明確不做的功能。

第二課

把需求翻成系統約束

產品語言要再壓一層，變成可估算、可驗證的工程條件。你不是只說「要即時」，而是要講清楚允許幾秒延遲、尖峰在哪、提醒多久內要送到。

即時報價

到底是 1 秒內、5 秒內，還是可接受 15 分鐘延遲？不同答案決定你要買什麼數據源、能不能便宜地做。

提醒 SLA

價格觸發後 5 秒內要通知，還是 30 秒內可接受？這會直接改變 alert engine 的掃描頻率與設計方式。

尖峰流量

開盤前 5 分鐘、開盤後 30 分鐘、重大消息時會不會瞬間暴增？容量預估必須先有假設，不然架構只是假精確。

資料保留

只存最新報價、存日線 K、還是要分時歷史？這決定你的儲存成本、索引與查詢策略。

從需求變成數字

例如「目標同時在線 5,000 人，自選股平均 20 檔，每 3 秒更新一次」，這種句子才足以推導頻寬與 fan-out 壓力。

從數字變成設計

當你知道提醒延遲要小於 5 秒，就能反推要不要用事件驅動，而不是每分鐘批次掃描一次。

第三課

畫出領域模型

很多系統一開始就亂，是因為名詞沒有定義好。資料模型不是表格細節，而是你對系統世界的切割方式。先把主要實體講清楚，API 和資料庫才會穩。

User

帳號、偏好設定、通知通道。

identity
Watchlist

使用者的自選股集合與排序。

portfolio view
Symbol

交易標的基本資訊，例如市場、代碼、名稱。

market catalog
Quote

最新成交價、買賣盤、漲跌幅等即時狀態。

realtime
Candle

K 線資料，決定圖表能回看多長時間。

history
AlertRule

提醒條件，例如大於某價格、漲跌幅超過多少。

rule
AlertEvent

規則被觸發的事實，通常要可追蹤、可稽核。

event
NotificationDelivery

提醒是否成功送達、失敗原因、重試次數。

delivery
MarketSource

外部行情來源的連線狀態、版本與延遲資訊。

source health
這一課真正的目的

避免把「提醒規則」和「提醒事件」混在一起。規則是設定，事件是系統觀測到的觸發事實，兩者不能混。

最常見錯誤

一開始只想 UI，不先定義實體，結果後面出現命名混亂、API 歪掉、資料欄位無限制地長。

第四課

設計資料流與系統架構

到這一步才開始畫服務。對第一版來說，我不會急著拆微服務，而是用一個穩的後端搭配背景工作，把資料流先設計清楚。你真正需要的是邊界，而不是服務數量。

1
Market Data Ingestion

連外部行情源，做標準化與更新頻率控制。

外部資料入口
2
Quote Cache / Stream

把最新報價放進 Redis 或其他快取層，供即時扇出使用。

即時層
3
API Backend

處理登入、自選股、提醒設定、歷史查詢與權限控制。

核心應用
4
Alert Engine

比對即時價格與使用者規則，產生可追蹤的 AlertEvent。

規則引擎
5
Notification Worker

根據事件送 Web Push、Email，並做重試與去重。

通知輸出
第一版推薦結構

Frontend + 單體 API Backend + Worker + Postgres + Redis + Queue。這樣已經足夠撐起很多中小型產品的第一階段。

為什麼先不要微服務

你現在最大的風險不是服務耦合，而是需求還沒穩。太早拆服務，會先把團隊速度拆掉。

第五課

先設計失敗場景

金融資訊類產品不能只設計 happy path。使用者不會因為你畫面好看就原諒錯價、重複通知、斷線後沒有提示。這些壞掉的地方，才是系統信任度真正的基礎。

外部行情源斷線

前端要標示 stale data，後端要記錄最後更新時間，告警系統要知道目前資料不可信。

WebSocket 重連

重連後要補一次最新快照，不然使用者看到的可能是空白或舊資料。

提醒重複發送

同一條規則在震盪區間可能連續觸發，要做去重、冷卻時間和事件 idempotency。

單一用戶規則過多

如果有人設 1,000 條提醒，你要限制配額，或重新設計 rule evaluation 的索引方式。

這一課的判斷標準

當系統壞掉時，使用者是否看得懂現在發生什麼事；當資料不可信時，畫面是否有能力誠實地說明狀態。

第六課

排開發順序與驗證方式

系統設計最後一定要落到開發順序。好的順序不是照技術層切，而是照價值鏈切。你要讓團隊每一階段都能交出一個更接近真實產品的版本。

Phase 1 帳號、自選股、Symbol 基本資料

先把使用者可以進入系統並建立自己的市場視圖。

Phase 2 行情導入與即時畫面

把最新報價流進系統，讓前端可穩定更新，不求華麗先求正確。

Phase 3 提醒規則與通知送達

完成產品真正的主價值，並開始量測提醒延遲和成功率。

Phase 4 歷史圖表、監控、營運儀表

補足觀測能力與使用體驗，讓產品進入可優化狀態。

你應該追的指標

報價更新延遲、提醒觸發延遲、通知成功率、WebSocket 連線穩定度、使用者日活與自選股數量。

你應該先驗證的事

使用者到底在意畫面更新速度，還是提醒的準確率。這兩者都重要，但優先級不能靠猜。

第七課

技術選型與擴充時機

技術不是先選，而是最後為需求服務。你要的不是最潮架構，而是在當前產品階段最可控、最容易維運、最容易交付的組合。

Frontend

可用 Next.js 或其他熟悉的 React 架構，重點是處理即時訂閱畫面、狀態同步與通知設定流程。

Backend

Node.js / NestJS 或 Go 都合理。選擇標準是團隊熟悉度、I/O 模式、維運成本，不是社群熱度。

資料與快取

Postgres 管使用者與規則，Redis 管最新報價與即時 fan-out，Queue 解耦提醒與通知。

何時再拆服務

當行情處理、提醒引擎、通知通道的負載或開發節奏明顯分離，再拆。不是因為「大家都這樣做」。

如果你現在要開專案

先交出一頁產品定義、一份 schema 草圖、一張資料流圖、一個 4 週 roadmap。這四份比先建 repo 還重要。

如果你要繼續下一步

就可以開始做 DB schema、API spec、事件流與監控設計。那時候你才是真的在做 implementation planning。

課後延伸

如果是我，下一輪會繼續想什麼

當以上七課都定下來，我下一步就不再談抽象概念，而是把這些內容壓成文件與介面。這也是系統設計真正開始影響實作速度的地方。

API Spec

例如 `/watchlists`、`/alerts`、`/quotes/snapshot`、`/notifications/preferences`，定義入參、回傳與錯誤碼。

DB Schema

把 `users`、`watchlists`、`alert_rules`、`alert_events`、`notification_deliveries` 真的畫成表。

Observability

定義哪些 log、metric、alert 要先上，不然系統壞了你只會看到使用者抱怨。

Cost Model

行情源成本、通知成本、即時連線成本。金融資料產品很容易不是技術先死，而是成本先失控。
