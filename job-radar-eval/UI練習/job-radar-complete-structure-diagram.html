Job Radar / Complete System Design

完整系統設計圖

這一頁不是零碎模組清單，而是把整個專案真正的系統設計關係畫出來： 從使用者入口、前端工作台、應用協調層、背景執行、資料管線、AI / RAG、到多資料庫與外部系統。 如果你要面試講解，這頁可以直接當主圖。

這頁在解釋什麼

你的專案不是單純爬蟲，也不是單純聊天機器人。它是一個以「市場快照」為核心的求職工作台： 前端承接搜尋與回訪，後端協調同步與非同步任務，AI 建立 grounded 問答與履歷匹配， 而資料層則分離產品主資料、執行期資料、歷史 archive 與 cache。

系統核心
Snapshot-Centered
執行模型
UI + Scheduler + Worker
AI 模式
LLM + RAG + Matching
資料分層
Product / Runtime / History

面試時的講法

最好的講法不是一開始就陷進 chunking 或 DB schema，而是先拿這張主圖講： 使用者問題、我怎麼拆成系統層次、AI 放在哪裡、 為什麼要有 background runtime、為什麼要多資料庫。

一句話總結

Job Radar 是一個模組化單體產品：以 Streamlit 工作台承接互動，以 crawl application service 協調資料刷新， 以 pipeline 和 analyzer 把 raw jobs 轉成 snapshot，以 AI layer 建立 grounded 問答與履歷匹配，以多資料層支撐持續運營。

02 / Entry Flow

前端入口與同步 / 非同步協調

這張圖專講使用者按下「開始抓取並分析」後，系統到底怎麼動。

Manual Query Flow
UI

使用者輸入條件

search_setup 收集角色、地區、關鍵字、自訂 query、頻率。

→
App Service

maybe_start_crawl()

先判斷 signature、saved search、快取是否可直接命中。

→
Branch A

有快照就同步回傳

直接 hydrate snapshot、cache views、渲染 overview / market。

Async Branch
Queue

建立 crawl job / signal

runtime DB 記錄 pending 狀態，UI 先回傳 staged loading。

→
Worker

lease job 並執行 pipeline

fetch、clean、analyze、persist、notification。

→
Finalize

UI 輪詢 signal

完成後 reload snapshot 並寫入 page context / saved search results。

為什麼這樣做

  • 同步入口盡量輕，避免 Streamlit request 被長任務卡住。
  • 第一次查詢可以走快取，未命中才建立 background job。
  • UI 不直接抓資料，而是透過 runtime DB 輪詢，這樣進度與失敗都可觀察。
  • scheduler 和 worker 分離，代表「決定要不要做」與「真正執行」是兩件事。
03 / Pipeline

抓取與資料處理管線

這張圖把從外部職缺站到市場快照的資料工程拆開來看。

Raw → Canonical → Snapshot
Step 1

Connector Fetch

各站點先拿搜尋結果，再視需要補 detail，輸出統一 JobPosting 模型。

→
Step 2

Cleaning / Canonicalization

canonical title / company / location / salary / url，並做 cross-source merge key。

→
Step 3

Dedupe / Merge

同來源 exact duplicate、跨來源同職缺合併、保留較完整的一筆。

→
Step 4

Analyzer

角色配對、relevance、技能 insight、task insight、source / role / location summary。

→
Step 5

Persist

寫入 jobs_latest.json、product_state、market_history、runtime signal。

為什麼 pipeline 不能直接輸出 raw jobs

  • 原始資料跨站格式不一致，直接用會讓 UI、AI、通知都變得脆弱。
  • 沒有 canonical normalization，跨來源同職缺無法合併。
  • 沒有 analyzer，AI 就只拿得到雜亂 description，很難做穩定回答。

這層也是 feature engineering

  • role / relevance / skills / tasks 都是 engineered signals。
  • 它們不是只給模型用，也給產品排序、統計、篩選與追蹤用。
  • 這也是為什麼這套系統比一般爬蟲更接近產品平台。
04 / AI

AI / LLM / RAG 系統設計圖

這張圖把 AI 層單獨拉出來，方便你面試時專講 retrieval 與回答品質。

Assistant Answer Flow
Input

Question

使用者問題與對話上下文。

→
Routing

Mode / Intent / Policy

判斷是 market、personalized、comparison、general，並決定 retrieval 與 token budget。

→
Knowledge

Chunk Build

從 snapshot、resume、market insight 建立 chunks，並用 chunk cache 減少重建。

→
Recall

Persistent ANN + Snapshot Candidates

長期索引候選召回，與當前 snapshot chunks 合併後去重。

→
Ranking

Hybrid Retrieval

embedding + lexical + signal + source bonus + market penalty + dynamic top-k。

→
LLM

Answer

以 grounded context 生成自然 QA 回覆。

Resume Intelligence
Parse

Resume Extraction

rule-based context selection + OpenAI profile summarization。

→
Match

Semantic + Exact Match

role similarity、exact overlap、embedding similarity、market fit。

→
Output

Gap / Recommendation

給 resume 頁與 assistant 個人化建議使用。

AI 層的關鍵設計

  • 不是所有問題都純聊天，大多數帶求職或時間脈絡的問題都會先走 retrieval。
  • chunking、top-k、rerank、ANN 不是分開的，而是同一條 answer path 的不同控制點。
  • AI monitoring events 讓你可以同時看 mode、latency、status、usage。

目前仍是產品型 AI，而非研究型大平台

  • 它強在 grounded answer、產品整合、維運與可解釋性。
  • 它還沒走到多 agent planner 或超大型全域知識編排。
  • 這是刻意取捨，目的是先把可靠與可用做出來。
05 / Storage

資料與儲存設計圖

這張圖不是 schema 細節，而是 ownership 與生命週期邊界。

產品主資料

  • product_state.sqlite3
  • saved searches、favorites、notification prefs、auth refs、AI monitoring。
  • 直接服務 UI 與產品邏輯。

執行期資料

  • query_runtime.sqlite3
  • crawl jobs、signals、leases、runtime queue。
  • 生命週期短，但對非同步協調至關重要。

歷史與快照

  • jobs_latest.json / snapshots / market_history.sqlite3
  • 服務當前市場檢視、RAG、歷史 archive 與未來趨勢分析。
Cache Layer

Embeddings / Resume / Runtime Cache

提升 LLM 與 retrieval 效能，但不應被當成主資料。需要固定 maintenance 與上限治理。

→
Maintenance Layer

runtime_maintenance / backup / restore drill

確保 cache 不膨脹、DB 可備份、狀態可觀測、系統可復原。

06 / Sequences

兩條最重要的 sequence

這裡是面試時最容易畫在白板上的版本。

手動查詢 sequence

User -> Streamlit UI -> crawl_application_service -> 快取 / signature / saved search 判斷 -> 若命中：直接回 snapshot -> 若未命中：建立 crawl job 到 query_runtime -> UI 先顯示 pending 狀態 Worker -> lease runtime job -> connectors / cleaning / analyzer -> persist snapshot + update signals UI -> 輪詢 runtime signal -> finalize 後重新載入 snapshot -> 渲染 overview / market / tracking

AI 助理 sequence

User Question -> assistant.service -> classify mode / intent / retrieval policy -> build chunks from snapshot / resume / insights -> persistent ANN candidate recall -> hybrid retrieval + rerank -> OpenAI responses.create -> write ai monitoring / usage / cache -> UI renderer -> grounded QA answer
07 / Why This Shape

為什麼系統會長成這個樣子

這裡講設計動機，不只是畫圖漂亮而已。

優點

  • 模組化單體比早期微服務更適合你目前的 stage。
  • UI / scheduler / worker 分離後，使用體驗與觀測性都更穩。
  • snapshot-centered RAG 讓回答更貼近「當前這批市場資料」。
  • 多資料層讓產品主資料、執行期資料、歷史資料不互相污染。

代價

  • 規則、routing、rerank 會逐漸累積，需要持續整理。
  • Streamlit 仍不是最理想的長期高互動前端框架。
  • 多個 SQLite 在早期很好用，但未來量級大了會遇到治理邊界。
  • AI 仍受限於模型 latency 與 answer-level evaluation 不夠完整。
