頁面類型

頁面類型是在回答「這頁的主要任務是什麼」。同樣都有按鈕和卡片，但目的不同，版面安排就會完全不同。

常見頁面類型

先說清楚是哪一種頁面，對方才知道你想做的是說服型、工具型、交易型，還是身份型畫面。

Page
Page Type / 頁面類型
Landing Page

介紹產品、活動或品牌，重點是吸引注意力與轉換。

Dashboard

快速看整體狀態、篩資料、做管理，重點是效率和掃描速度。

Settings Page

修改帳號、偏好或系統參數，重點是分類清楚與表單穩定。

Detail Page

聚焦單一項目，像商品、文章、案件或任務，重點是理解內容與做下一步。

Checkout Page

交易最後一步，重點是減少分心、清楚顯示步驟與費用。

Auth Page

登入或註冊頁，重點是簡潔、信任感與低阻力。

Onboarding Flow

引導新手一步一步完成設定，重點是資訊量控制和節奏。

Profile Page

展示個人資料、作品、活動或偏好，重點是資訊可讀與入口清楚。

Navbar

Sidebar

Sticky Header / Footer

改 UI 時，最常改的是這些層

你可以把參數分成幾層來想：顏色、字體、間距、尺寸、邊框、陰影、位置、動效、響應式。這樣溝通時不會只說「幫我改漂亮一點」。

UI 注意事項

這些不是元件名稱，而是做 UI 時最常踩到的實戰重點。你越早理解，畫面會越穩。

Visual Hierarchy

先讓人看到最重要的東西，再看到次要資訊。

Guide
Priority / 視覺層級
為什麼重要

主標、主按鈕、關鍵數字不能跟次要資訊一樣重，不然使用者會失去焦點。

Spacing Scale

間距要有規則，不要每個地方都亂跳。

Guide
Spacing / 間距
你可以這樣講

請統一用 8 / 12 / 16 / 24 這種 spacing scale，不要每張卡都用不同內距。

Alignment & Grid

元素要有共同對齊基準，畫面才不會散。

Guide
Alignment / 對齊
為什麼重要

標題、按鈕、卡片、欄位如果沒有對齊，即使顏色很好看，畫面也會顯得亂。

Consistency

同一種功能要用同一種樣式和互動邏輯。

Guide
Consistency / 一致性
你可以這樣講

主操作都用 primary button，危險操作都用 danger style，不要每頁都重新發明一次。

Contrast & Readability

再好看的 UI，如果看不清楚，就不好用。

Guide
Readability / 可讀性
為什麼重要

文字太淡、字太小、背景太花，都會讓資訊讀取成本變高。

Interactive States

元件不是只做 default，還要補齊互動狀態。

Guide
States / 狀態
你可以這樣講

這組元件請補完整 state：default、hover、focus、disabled、loading、error。

Responsive Design

桌面版能用，不代表手機版自然就好用。

Guide
Responsive / 響應式
為什麼重要

多欄桌面版到手機常要改成單欄，固定區塊、表格和點擊區都要重新想。

Accessibility Basics

不能只讓看得到、用滑鼠的人能操作。

Guide
A11y / 無障礙
你可以這樣講

保留 focus ring、不要只靠顏色表達狀態、互動元件要能鍵盤操作。

Tailwind CSS

不是固定外觀，是組裝式風格。

MUI

完整元件庫，預設質感一致。

Pure CSS / Design Tokens

只改外型時，通常先不用加套件。

Package
What It Gives / 提供內容
內容

CSS variables、色彩、字體、圓角、陰影、間距、排版規則。

適合情境

只是改外型、頁面不多、想完全自訂風格。

Tailwind CSS

Utility-first 樣式工具，不是完整元件庫。

Package
Utility / 原子化樣式
內容

大量 class 組合間距、顏色、字級、grid、breakpoints、hover/focus。

適合情境

想快速切版又不想被固定外觀綁住。

Bootstrap

傳統完整型 CSS Framework。

Package
Framework / 完整框架
內容

Grid、按鈕、表單、Navbar、Modal、表格等很多都能直接用。

注意

外觀辨識度高，追求強品牌感通常要再蓋很多樣式。

Headless UI / Radix UI

提供互動邏輯和 accessibility，但幾乎不帶固定外觀。

Package
Behavior Layer / 行為底座
內容

Dialog、Menu、Popover、Tabs、Select 的鍵盤操作、焦點管理與 aria 行為。

適合情境

想保有自訂視覺，但不想自己重做複雜互動。

MUI / Ant / Chakra

完整 React 元件庫，開發速度快。

Package
Component Library / 元件庫
內容

大量現成元件 + theme system，適合 dashboard、後台、商業產品。

注意

客製自由度比自己寫 CSS 低，但速度通常更快。

Icons / Motion Packages

這類是補強型套件，不是完整 UI 架構。

Package
Add-ons / 補強套件
內容

像 Lucide、Font Awesome、Framer Motion，分別補圖示與動效。

適合情境

你只缺圖示或動畫，不缺主要 UI 系統。

下一步學習

如果前面這些你都已經懂了，接下來應該學的是「讓 UI 真正能落地、能維護、能交接」的東西。

Design Tokens

把顏色、字級、圓角、陰影和間距變成一套規則。

Next
System / 系統化
這是在學什麼

不要每次都直接填色碼和像素，而是定義 `brand / surface / text / spacing / radius` 這些 token。

Variants & Component API

一個元件通常不只一種長相，要先定義 variant。

Next
Component API / 元件規格
你可以這樣講

請整理成 variant system，至少定義 tone、size、icon、disabled、loading。

Real Content & Edge Cases

不要只用漂亮假資料，要考慮真實內容會怎麼把版面撐壞。

Next
Edge Cases / 邊界情況
這是在學什麼

長標題、超短名稱、沒有頭像、沒有資料、數值異常，都會影響 UI 是否穩定。

Localization & i18n

換語言後，長度、格式與閱讀順序都可能改變。

Next
Localization / 在地化
這是在學什麼

按鈕長度、日期格式、貨幣、姓名順序都要依 locale 思考。

Data Formatting

數字、日期、金額和單位的顯示方式很重要。

Next
Formatting / 資料格式
你可以這樣講

這區的數字請統一格式，金額顯示幣別，時間要決定用相對時間還是絕對時間。

Density & Scanability

資訊多的畫面，不是越塞越好，要讓人掃得動。

Next
Density / 資訊密度
這是在學什麼

後台、表格、資料卡很多時，要決定是 compact 還是 comfortable。

Handoff & Specs

畫面好看不夠，還要能被工程正確做出來。

Next
Handoff / 交接
你要補什麼

尺寸、間距、state、responsive 行為、空資料與錯誤情境都要寫清楚。

Design QA

最後一關不是再改風格，而是檢查有沒有偏掉、漏掉、對不起來。

Next
QA / 設計檢查
檢查方向

Spacing、字級、對齊、hover / focus、手機版、空狀態與資料溢出情況。
