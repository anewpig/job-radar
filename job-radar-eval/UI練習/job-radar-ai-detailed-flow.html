AI Flow Handbook

Job Radar AI 詳細流程圖與執行解剖

這份頁面只講 AI。本頁把整套 AI 執行鍊拆成三條主線：履歷分析、RAG 市場助理、AI 監控與 fallback。 每一段都對齊目前程式碼的真實流程，並補上設計理由、優點、缺點、替代方案，以及未來擴充時應該怎麼切。

這頁回答什麼

  • AI 在哪幾個點真的介入，不是只有「有接 OpenAI」。
  • 履歷分析為什麼先做 rule-based，再讓 LLM 疊上去。
  • RAG 助理怎麼從市場快照切 chunk、檢索、組 prompt、產生回答與引用。
  • 沒有 API key、模型失敗、缺履歷資料時，系統怎麼退回 fallback。
  • 哪裡有 cache、哪裡有 token 成本、哪裡有監控事件。
  • 如果之後要上更多模型、更多資料源、向量資料庫、背景 worker，要從哪裡切。

02 Resume Flow

履歷分析詳細流程圖

這條流程的核心原則是「先保證可用，再追求更準」。所以不是直接把整份履歷丟給 LLM， 而是先用規則抽出一個 fallback profile，再讓 LLM 在這個骨架上做修正與補全。

1

接收履歷輸入

支援上傳檔案與手動貼文字，先做文字擷取與內容合併；若抽不出文字就直接停止。

PDF / DOC manual text
2

Rule-Based 初稿

用關鍵字、技能類別、任務 pattern、領域 pattern 抽出 target roles、skills、tasks、summary。

永遠可用 低成本
3

若有 API key，才把清理後履歷重點與 fallback brief 一起送去 Responses Parse，產出結構化 payload。

responses.parse schema output
4

合併成 ResumeProfile

LLM payload 不是覆蓋 fallback，而是合併。缺的欄位回退到 fallback，確保 profile 完整。

merge notes
5

先做規則匹配，再選候選職缺進行 title LLM scoring 與 embedding scoring，最後寫回 session / store。

文字輸入 上傳檔與貼上文字都先合併成單一 resume_text，來源名稱另外保留。 把後面 extractor 的輸入介面單純化，避免每個分析器都分別處理檔案格式。 流程乾淨、容易測試、UI 可給 staged feedback。 若前面 OCR / 文字擷取失敗，AI 層根本拿不到資料。 直接讓 LLM 讀原始 PDF；但成本高、穩定度差，也更難做 fallback。 如果要支援掃描 PDF，可在這一層前插 OCR worker，不要把 OCR 邏輯塞進 ResumeAnalysisService。
規則抽取 先跑 RuleBasedResumeExtractor，用 role keywords、skill categories、task patterns、domain patterns 建初稿。 先把最穩定、可解釋、零外部依賴的能力放前面。 便宜、快、可離線、失敗率低、方便 debug。 對語意理解有限，對跨句推論與抽象經驗總結較弱。 一開始就全靠 LLM。缺點是沒有 fallback，還會把穩定性完全綁在模型上。 可把規則抽取再拆成 skill extractor、role inference、task parser 三個子模組，讓維護更細。
LLM 擷取 只有在有 API key 且 schema 可用時才啟用 OpenAIResumeExtractor；而且會先查 resume_profile_extracts cache。 LLM 擅長補語意與結構化摘要，但不該每次重算。 抽得更像人話，target roles 與 prompts 通常更準。 仍有延遲與成本，也可能回傳不可解析內容。 本地模型或小型分類器。可降成本，但精度與繁中履歷品質未必穩。 若未來量大，可把 profile extraction 改成背景 job 並加版本化 cache key。
Profile 合併 LLM payload 與 fallback profile 合併；summary、roles、skills、tasks、keywords 取 union / fallback。 讓 LLM 成為 enhancement，而不是 single source of truth。 大幅降低 LLM 漏欄位時的破壞性。 有時會把 fallback 的噪音也一起帶進去。 完全信任 LLM payload；但一旦 schema 漏欄位，整份 profile 就殘缺。 之後可引入欄位級 confidence，決定某欄是保留 fallback 還是採 LLM。
職缺匹配 先跑 ResumeMatcher 算全量規則分，再選候選職缺做 title LLM + embedding semantics，最後 merge 回原排序。 控制成本，不把所有 jobs 都送 LLM / embeddings。 全量有 baseline、局部有高精度、成本比全量 AI 低很多。 候選剪枝若太 aggressive，可能錯過低 baseline 但高語意匹配的職缺。 所有職缺全量 embedding / rerank。更準，但成本和延遲更高。 量大時可把初篩改成 ANN / vector index，再接 reranker，而不是只靠 fallback 分數挑候選。
03 Match Pipeline

履歷匹配內部評分流程

履歷匹配不是一個單分數，而是四層疊加：職稱、技能語意、任務語意、關鍵字 / 領域。 其中 exact hit 會當 bonus 疊上去，最後才形成 overall score。

A

Fallback 全量初篩

規則 matcher 先對全部 jobs 做 role / skill / task / keyword / exact match 打分。

B

候選職缺挑選

從 fallback top results 挑一小批 candidate；職稱 exact hit 或 role similarity 很高者也會被補進來。

C

Title + Semantic AI Scoring

標題相近度走 Responses Parse 批次打分，技能 / 任務 / keyword 相似度走 embeddings cosine。

D

Merge 回完整排序

只把 candidate jobs 的結果替換成 LLM 版，其餘 jobs 保留 fallback 版，最後整體再排序。

為什麼不全量送 LLM

  • 職缺數一多，title scoring 與 embeddings 會直接放大延遲與成本。
  • fallback matcher 其實已經能把大部分完全不相干的職缺排掉。
  • 先 coarse ranking，再對前段重算，是一種典型兩階段檢索 / rerank 方案。

目前成本控制點

  • LLM 候選數有限制，不是全部職缺都會進 AI scoring。
  • Title similarity 與 embeddings 都有檔案 cache。
  • 若 role 已 exact match，會直接跳過 title LLM。
  • 若沒有 API key 或模型失敗，整條鏈直接退回 rule matcher。
履歷匹配評分結構
分數層 來源 用途 現在這樣做的好處 限制 如果升級
Title Similarity Responses Parse 批次對 target_roles × job.title 打 0~1 分。 修正純規則職稱比對太硬的問題。 對「AI應用工程師 vs AI工程師」這種相近稱呼更有彈性。 仍然只看 title / matched_role，不看整個 JD。 可換成 cross-encoder reranker，或引入 title normalization service。
Skill Similarity 履歷 skills / roles 與 job requirement / extracted skills 做 embedding cosine。 補語意相近但字面不同的技能。 比單純 keyword overlap 更能理解變體說法。 embedding 仍不等於真正的 job fit。 可加入 skill ontology 或 skill graph，讓語意相似有更可控的權重。
Task Similarity 履歷 preferred tasks / prompts 與 job work content / summary 做 embedding cosine。 讓工作內容匹配不只看技能。 對 PM、應用工程師這類任務導向角色特別重要。 若 JD 工作內容寫得很空泛，資訊品質會差。 可引入任務 taxonomy，先標準化 task label 再比。
Keyword / Exact Hit 關鍵字命中、精確技能命中、精確任務命中、精確職稱命中。 保留高可解釋性訊號，讓分數能回推原因。 使用者容易理解「為什麼這份履歷適合這個 job」。 對語意改寫較不敏感。 可改成 feature store，統一把 exact / semantic / behavioral features 做線性或學習式融合。
04 RAG Flow

AI 助理 / 市場問答詳細流程圖

主線仍然是 snapshot-centered RAG：回答依賴這次搜尋抓回來、清洗完、分析完的市場快照，而不是整個歷史世界。 另外現在 general_chat 也會保留市場檢索當語境；若是具體職缺薪資題且該職缺沒有真實薪資， 則會在 retrieval 後再接一條薪資預估支線。

1

問題進站與 mode routing

先判斷這題要走 market_summary、personalized_guidance、job_comparison 還是 general_chat，再把最近最多 2 輪問答摘要合進 retrieval query。

answer_mode 2-turn context
2

從 snapshot / resume 建 chunks

把 jobs、salary、skills、work content、market aggregate，以及履歷 facets + resume windows 轉成結構化 knowledge chunks，而不是直接把整份資料庫丟去做向量搜尋。

job-summary resume-window
3

Hybrid Retrieval

先做 coarse candidate selection，再用 embedding、lexical overlap、signal overlap、source type bonus、role alignment、job reference bonus 與 market penalty 混合排序。

為什麼用 Market Snapshot 做 RAG

因為這個產品要回答的是「這次搜尋結果代表的市場」，不是做職涯百科。用 snapshot 做上下文，回答才會和畫面上的職缺、技能洞察與工作內容對得起來。

general_chat 現在的邊界

general_chat 不再等於完全不檢索。它會保留市場脈絡與可選外部查詢，但 prompt 明確禁止硬把所有問題拉回求職市場。

這樣做的限制

回答仍以當前 snapshot 為核心。如果使用者問的是跨多次搜尋、跨月份或跨產業長期趨勢，仍需要獨立 historical retrieval 層，而不是直接混進目前 hot path。

薪資預測支線現在怎麼掛進來

這條線只處理「具體職缺」且「原始職缺沒有薪資」的情況。若原職缺已有真實薪資，就直接優先顯示真實資料； 若是「AI 工程師市場薪資大概多少」這種市場題，仍維持 RAG-only，不會硬做泛化預測。

05 Retrieval Deep Dive

RAG 檢索器怎麼挑資料

這套檢索器不是單純 embedding nearest neighbor。它會先從問題抽 signals / intents，再把 chunk 類型、角色對齊、 job reference、aggregate/specific-job 差異一起考慮，目的是讓答案更像「產品化市場助理」，而不是只會把語意相近的字串拉上來。

問題先做哪些理解

  • 抽取 signals：例如 python、rag、salary、location、role。
  • 判斷 intents：例如 skill_gap、work_content、salary、market、source_distribution。
  • 判斷是不是 aggregate 問題，像「目前常見技能」會優先偏向 market aggregate chunks。
  • 判斷是不是 specific job 問題，像「這個職缺主要做什麼」會壓低 market chunks 的權重。
  • comparison 題還會額外做 comparison coverage，避免只檢到其中一邊。

Hybrid score 真正考慮什麼

  • embedding：0.52，提供語意近似底盤。
  • lexical overlap：0.18，補字面命中。
  • signal overlap：0.22，補技能 / 類型訊號。
  • source type bonus：依 intent 加成最適合的 chunk 類型。
  • market priority bonus：把 occurrences / importance 轉成市場型 chunk 的偏好。
  • role alignment / job reference bonus：如果問題和 chunk 指向同一角色或同一職缺，會再加權。
  • specific-job market penalty：問具體職缺時，避免 market-skill / market-task 把 job-detail 蓋掉。
Chunk 類型與用途
Chunk 類型 來源 適合回答什麼 為什麼要保留 問題點
job-summary 每筆職缺的 title / company / source / summary / skills 職缺比較、個別 JD 參考、來源例子 最通用，是其他細類型 chunk 的保底證據 資訊粒度混雜，不一定是回答某個子題最精準的來源
job-salary 有薪資資訊的職缺 薪資區間、薪資揭露情況 把薪資問題從一般 JD 摘要拆出去，避免被其他訊號淹沒 很多平台本來就沒薪資，coverage 不完整
job-skills required skill items / extracted skills 技能需求、技能缺口 讓技能回答更具體，不只停在 summary 文字 技能欄位品質很吃 connector 與清洗規則
job-work-content work content items 工作內容、任務型問題 支援「這個職位平常在做什麼」類問題 若來源站點切段不穩，內容粒度可能很不一致
market-skill-insight 市場技能聚合結果 常見技能、核心技能、優先補強 能回答「整體市場」而不是只回答單筆職缺 失去職缺細節，需要配 citation 才不會太抽象
market-task-insight 市場工作內容聚合結果 常見工作內容、職責分布 符合產品定位，讓助理更像市場分析器 統計方法若太粗，會把不同細任務混在一起
resume-summary 履歷摘要、facets、resume windows 個人化建議、技能缺口、投遞建議 已不再只有單塊 summary，能提供比較細的個人背景證據 還沒有做到 project / accomplishment 等更細層級

為什麼不是純向量檢索

純向量很容易把「語意像但資料型態不對」的 chunk 拉上來，例如問薪資卻拿到技能摘要。source type bonus、job reference bonus 與 market penalty 就是為了修這個問題。

為什麼要做 candidate 粗篩

全量 chunks 一多，全部做 embedding + hybrid rerank 的成本會膨脹。先粗篩候選，再精排，延遲和成本都比較可控。

citation 為什麼獨立選

最適合生成答案的 chunk，不一定最適合拿來展示證據。citation selection 應視為獨立步驟，這樣才能同時照顧回答可讀性與證據可查性。

06 Answer Modes

AI 助理的四種回答模式

這裡沒有先做超複雜的 planner，而是先用輕量 mode routing 把回答風格、retrieval 策略與 section builder 收斂住。 值得注意的是：general_chat 不代表完全不檢索，只是 prompt 會明確禁止硬把問題拉回求職市場。

market_summary

預設模式。適合問「目前常見技能是什麼」「這批職缺重點是什麼」。輸出會偏市場分布、核心技能、薪資樣態、趨勢提醒。

personalized_guidance

只有在有 resume_profile 且問題含「我 / 我的 / 履歷 / 適合我」等線索時才走。輸出會偏缺口、優先補強、投遞建議。

job_comparison

問題裡有「比較 / 差異 / vs / 哪個比較適合」等線索時走。除了檢索外，還會額外做 comparison coverage，保證比較對象都被帶進來。

general_chat

當問題不直接屬於求職主線時走這條。它仍可補市場檢索與可選外部查詢當語境，但回答會偏自然 QA，不會強制套用市場分析格式。

這種 mode routing 的設計取捨
面向 目前做法 優點 缺點 替代方案
路由邏輯 用 heuristics 判斷 comparison / personalized / market / general_chat。 成本低、可預期、快、好 debug。 語言表達一變，仍可能誤判 mode。 可改成小模型 classifier，或拆成 subject_mode / tone_mode / retrieval_mode 三層。
回答格式 每種 mode 固定自己的 section builder。 前端展示穩、容易做結構化 UI。 自由度較低，對複合問題比較僵。 可改成更通用的 section schema，再由前端動態 render。
上下文來源 主要來自 snapshot、optional resume profile，以及 general_chat 的語境補充。 邊界清楚、易引用。 無法自然回答跨 snapshot 的長期趨勢。 可加歷史層 retrieval，但必須把資料層來源清楚標示。
07 Monitoring & Fallback

AI 監控與 Fallback 流程圖

這套系統的 AI 不是「要嘛成功、要嘛爆掉」。它刻意把 blocked、fallback、error 分開記錄， 目的是讓你看得出來哪種失敗是因為缺資料、哪種是模型錯、哪種只是沒配 API key。

1

前置條件檢查

例如問個人化問題卻沒有 profile，或沒有 API key 時，就不會硬打模型，而是先走 blocked / fallback。

2

執行 AI 或 Rule Path

履歷分析會先建立 fallback，再嘗試 LLM；RAG 助理則若能執行就直接跑，失敗才記 error。

3

記錄 usage / latency

把 requests、token、cached_input_tokens、latency_ms、model_name、query_signature 都整理進 metadata。

4

寫入 ai_monitoring_events

最後統一寫進 ai_monitoring_events，讓後端控制台能看成功率、模式分布與延遲。

5

UI 呈現結果

成功就展示答案 / 報告 / matches；blocked 就給引導；error 就顯示格式化錯誤訊息。

記錄履歷抽取是否成功、用哪個 model、用了多少 token、抽到多少 roles / skills。

resume.match_jobs

記錄匹配花多久、候選 jobs 數量、matches_count、top_score、scoring_method。

resume.analyze_resume

整條履歷流程的 end-to-end 監控，方便看整體體驗而不是只看單步。

assistant.answer_question

記錄模式、used_chunks、citations_count、history_turns_used、是否 blocked_missing_profile。

assistant.generate_report

記錄報告生成是否成功、用了哪些 chunks、是否有履歷背景。

status 分類

至少分成 success、error、blocked_missing_profile，不是單純成功 / 失敗二分法。

這種監控設計的優點

  • 可以把產品問題、資料問題、模型問題分開看。
  • 可以直接做 latency budget、mode usage、成功率趨勢。
  • 因為 query_signature 有存，後面追特定搜尋條件也比較容易。

現在的限制

  • 監控目前主要是 event log，不是完整 trace。
  • 沒有 span-level breakdown，無法直接看 retrieval 花幾毫秒、LLM 花幾毫秒。
  • 沒有自動告警，需要人去後台看 summary。
08 Cache & Cost

Cache、成本與穩定性控制

目前這套 AI 架構之所以撐得住，是因為它不是每個步驟都即時計算。重複可重現的 AI 中間結果，大多有檔案 cache。

resume_profile_extracts

同一份整理後履歷上下文、同一模型、同一 role hint，不需要反覆打履歷結構化抽取。

resume_match_titles

職稱比對結果可重用，尤其同一份履歷反覆開同一批 jobs 時，title LLM 不用重跑。

resume_match_embeddings

履歷匹配時技能 / 任務 / keyword embeddings 會存檔，避免每次匹配都重新 embed 文本。

rag_embeddings

RAG 檢索 query 與 candidate chunks 的 embedding 會存檔，降低 repeated question 成本。

Usage Metadata

所有主要流程都把 request / token / cached_input_tokens 帶進 monitoring metadata，方便之後做成本統計。

Fallback Notes

履歷 profile 上的 notes 會把「沒有 API key」「LLM 失敗已改用規則分析」直接記下來。

目前成本控制與可替代方案
控制點 現在做法 優點 缺點 替代方案
模型呼叫次數 只在必要步驟打模型，履歷匹配也只重算候選區。 成本受控。 候選外的 job 不會得到 AI 強化分數。 可改用更便宜的 reranker 當第二階段。
embedding 重複計算 文字 hash 後做檔案 cache。 很適合目前單機架構。 多機部署時 cache 不共享。 可換成 Redis / object storage / vector service cache。
答案長度 回答模型有固定 temperature / max_output_tokens。 可預估成本與格式。 對複雜問題可能太短。 可依 answer_mode 動態調整 token budget。
錯誤容忍 履歷分析有 rule fallback，助理側則記 error 或 blocked。 核心功能不中斷。 RAG 助理目前沒有完整 answer-level fallback 模型。 可加 second model fallback 或 templated fallback answer。
09 Design Decisions

每個關鍵設計點，為什麼這樣做

這一段不是只列優點，而是把設計選擇的成本一起說清楚。你之後如果要繼續演進，就要從這些決策點下手，而不是從 UI 表面改。

選擇一：履歷分析採 fallback-first

理由：這個功能屬於高頻互動，且沒有 AI 時也應該能跑。

優點：可用性高、成本穩、出錯可回退。

缺點：架構比「全靠 LLM」更複雜，還要維護規則層。

替代：全 LLM pipeline，或改用傳統 NLP classifier。

選擇二：RAG 以 market snapshot 為中心

理由：產品核心是分析本次搜尋市場，而不是做職涯百科全書。

優點：引用清楚、答案可驗證、與畫面一致。

缺點：知識範圍受限，長期歷史問題回答能力弱。

替代：全庫向量檢索，或 snapshot + 歷史雙層檢索。

選擇三：混合檢索，不只靠向量

理由：求職問題很吃資料型態，單靠向量容易把類型錯的 chunk 拉上來。

優點：更能控制「薪資問薪資、技能問技能、來源問來源」。

缺點：規則多，調參成本高。

替代：純 vector search + reranker，或 learning-to-rank。

選擇四：事件式監控，不先上 tracing

理由：目前是單機產品，先把產品層 KPI 看清楚比先上複雜 tracing 更值。

優點：實作成本低，已足夠看成功率與使用模式。

缺點：無法精準拆每個內部步驟耗時。

替代：OpenTelemetry span tracing + metrics dashboard。

選擇五：檔案 cache 而非外部 cache service

理由：目前部署型態單純，檔案 cache 寫法直觀又便宜。

優點：實作快、可離線查看、沒有外部依賴。

缺點：不適合多機、不易做淘汰策略。

替代：Redis、S3、共用向量服務、SQLite cache index。

選擇六：UI 端直接驅動 AI 執行

理由：Streamlit 互動以同步請求最直觀，產品開發速度快。

優點：回饋快、流程清楚、容易做 staged status。

缺點：長任務容易卡互動，橫向擴展有限。

替代：改成 background AI jobs + polling，前後端解耦。

10 Expansion Playbook

如果要擴充，應該怎麼拆

下面這段不是抽象建議，而是按現在程式結構給你具體切法。重點是：不要一次把整套 AI 全重寫，應該按瓶頸拆。

要加更多模型

先把 extractor / matcher / assistant 的 model policy 抽成 provider config layer。保持 service API 不變，先替換底層 provider。

要做多模型 fallback

把目前單一 OpenAI path 改成 primary / secondary provider chain，監控事件要多記 provider 與 fallback_level。

要做長期知識庫 RAG

新建 historical chunk builder 與 retrieval namespace，不要直接把歷史資料混進現在的 snapshot chunks。

要做向量資料庫

先替換 EmbeddingRetriever._embed_texts 的 cache / storage，再逐步把 chunk index 從檔案 / memory 遷到 vector DB。

要降低同步等待

把履歷分析與報告生成改成 background jobs；前端保留 status panel，但改 polling job state，而不是直接同步等待模型回覆。

要做品質評測

把 assistant.answer_question 的輸入與輸出做離線評估集，建立 correctness / citation quality / usefulness 三種評分。

常見擴充情境與建議切點
擴充需求 先改哪裡 不要先做什麼 建議的遷移順序
更多 AI 問題量 assistant service、retrieval cache、monitoring dashboards 不要第一步就整套改成微服務 先 cache / budget → 再 background jobs → 最後才考慮服務拆分
更多履歷上傳量 resume extraction queue、OCR stage、profile cache versioning 不要把 OCR 混進現有 service 同步流程 先抽 pre-processing → 再 job queue → 再 provider abstraction
更準的匹配排序 candidate selection、reranker、feature fusion 不要直接把所有 jobs 全量送大模型 先強化候選挑選 → 再加 reranker → 再做學習式排序
跨快照 / 長期趨勢問答 歷史 chunk layer、source attribution、answer UI 不要直接把所有快照灌進同一 prompt 先建立歷史索引 → 再做 dual retrieval → 再調整引用 UI
企業化治理 provider policy、PII masking、audit logs、RBAC 不要只補前端權限提示 先資料治理 → 再 provider routing → 再審計報表

09 LLM Tuning

如果後續要把 LLM 調得更細，我會怎麼做

先講結論：這套系統目前最值得優化的，不是第一時間就做 fine-tuning，而是先把 prompt stack、context budget、mode routing、guardrail、fallback、評測集 定穩。因為這些會同時影響答案品質、延遲、成本與可維護性。只有當你已經知道「問題主要出在模型本身」，才值得再往 supervised fine-tuning 或 preference tuning 走。

第一層：先調 prompt stack

先把 system instruction、answer mode instruction、context injection、citation rule、輸出 schema 分開管理，別把所有規則塞進同一段長 prompt。

第二層：再調 context budget

控制歷史對話、resume context、retrieved chunks 的 token 預算，避免把重要 chunk 擠掉，或讓 prompt 變成昂貴但沒重點的大雜燴。

第三層：最後才考慮 fine-tuning

若 answer mode、prompt、retrieval、citation、fallback 都已穩定，但特定風格或判斷仍長期失真，再評估 fine-tuning，而不是把它當第一招。

調校面向 這個專案該先看什麼 為什麼重要 實際做法
System Prompt 角色定位是否清楚區分市場摘要、個人化建議、職缺比較 如果 system 層太模糊，後面加再多 retrieval chunks 都會被模型用錯方式解讀 把每個 answer mode 的核心任務、禁止事項、輸出欄位拆成獨立模板
Answer Mode Routing 分類器是否常把問題送錯模式 很多「答案不好」其實不是模型差，而是走錯 pipeline 先建立 mode confusion matrix，再針對高混淆題型補 routing 規則或 few-shot examples
Context Ordering resume、market summary、retrieved chunks 的排列順序 模型通常對前段與最後 constraint 更敏感，順序錯會讓它忽略真正關鍵資料 把 instruction 放前、task-specific context 放中、輸出格式與 citation rule 放後
Output Schema section label、citation slots、comparison slots 是否穩定 如果 schema 常變，你的 UI、後處理、評測都會不穩 優先讓回答變成半結構格式，再由 parser 組回 UI sections
Guardrails 是否亂補薪資、亂推論雇主政策、亂延伸履歷背景 求職場景很怕 hallucination，尤其是薪資、要求、匹配理由 在 prompt 明文禁止「未被 chunk 支持的事實」，並在 citation selection 再做二次保護
Temperature / Sampling 摘要、比較、個人化建議的穩定度 這類產品大多偏分析型，不需要很高創造性 市場摘要與比較題建議走低溫；只有鼓勵式 guidance 才略放開

Prompt Stack 應該怎麼拆

  • Layer 1：全域 system prompt，定義角色、資料邊界、禁止幻覺。
  • Layer 2：answer mode prompt，定義市場摘要 / 個人化 / 比較的任務。
  • Layer 3：context composer，把 resume、snapshot、retrieved chunks 組裝成有順序的上下文。
  • Layer 4：output schema，固定 section labels、citation 規則、字數上限。
  • Layer 5：post-processing policy，決定哪些 section 缺失時要補 fallback。

這樣拆的優點

  • 問題比較容易定位，知道是 routing、context 還是 generation 出錯。
  • 可對不同模式做 A/B prompt 調校，而不會牽動整套 assistant。
  • 更容易做 regression snapshots，因為 prompt 變動點清楚。
什麼情況才真的該做 fine-tuning？

在這個專案裡，只有當你已經做完以下事情，卻仍長期遇到同類型失真，才建議進一步 fine-tune： 第一，你已經有穩定的 answer mode routing；第二，你已有固定的 prompt templates 與 evaluation set；第三，你已經確認 retrieval quality 沒有明顯問題； 第四，你要優化的是 語氣、格式一致性、特定分類判準、特定段落結構 這類「模型習慣」問題，而不是知識缺失問題。 如果根本問題是檢索不到對的 chunk，或 resume context 太亂，那 fine-tuning 幾乎一定不是最值得先做的投資。

10 RAG Tuning

如果要把 RAG 調得更細，我會怎麼拆

RAG 調校不要只盯著 embedding model。真正影響最大的，通常是 chunk 長什麼樣、metadata 怎麼帶、query 怎麼改寫、candidate 怎麼粗篩、rerank 權重怎麼配、citation 怎麼選。 這套 Job Radar 的優勢是本來就不是純向量檢索，所以很適合往更細的 hybrid retrieval policy 繼續調。

Chunk 調校

先調 chunk family、chunk 大小、chunk 內容密度，而不是先換模型。

Retrieval 調校

把 coarse filter、hybrid weights、source type bonus、role alignment bonus 分開看。

Citation 調校

回答能生成，不代表引用就對。citation selection 應被視為獨立步驟，而不是 retrieval 排名的附屬品。

RAG 面向 現在這個專案該怎麼細調 優點 風險 / 代價
Chunk Granularity 把 market summary、skill insight、task insight、job detail、resume summary 拆成更明確的 namespace 能降低「問技能卻拿到 job summary」這類型別錯配 chunk 種類一多，檢索規則與 metadata 管理也會變複雜
Chunk Window / Overlap 針對長 JD 或長 work content，測試 200 / 350 / 500 token chunk 與 overlap 差異 可減少關鍵 requirement 被切斷 chunk 量增加會拉高 index 與 rerank 成本
Metadata Filters 依角色、來源、地區、salary availability、freshness 建立 filter 能先把不相干 chunk 擋掉，提升 top-k 質量 filter 太硬會降低 recall
Query Rewrite 針對 follow-up question，把 conversation context 摘成 retrieval query，再拆出核心意圖 多輪問答時 recall 更穩 rewrite 出錯會把檢索整體帶偏
Hybrid Weights 把 embedding、lexical、source bonus、market priority、role alignment 做實驗表 比盲調單一 cosine threshold 更可控 需要 evaluation set，不然只是感覺調權重
Reranker 未來可加 cross-encoder 或 model-based reranker，但先只 rerank top candidates precision 提升通常很明顯 延遲與成本會增加，需要先控 candidate pool

Chunking 最值得先做的三件事

  • 把 job detail chunk 和 market aggregate chunk 分 namespace，避免混在一起。
  • 對 resume summary chunk 加上顯式 metadata，例如 is_personal_context、role_targets。
  • 對每個 chunk 記錄 freshness、source_type、importance，讓 retrieval policy 可觀測。

Retrieval 最值得先做的三件事

  • 建立 query type → chunk type 的 expectation 表。
  • 把 top-k candidates 與最終 citations 都記錄到 monitoring event。
  • 把「沒檢到好 chunk」和「有檢到但回答用錯」分開統計，否則你不知道該調 retrieval 還是 generation。
如果未來要上 vector DB，順序應該怎麼走？

正確順序通常不是「先上向量資料庫再說」，而是： 第一步，先把 chunk schema、metadata、namespace、evaluation set 定穩； 第二步，把目前 cache / embedding layer 改造成可抽換 index provider； 第三步，先讓 historical retrieval 或大流量場景接 vector DB； 第四步，再考慮把 snapshot retrieval 主路徑遷過去。 這樣做的好處是，你是在已知檢索策略有效的前提下搬家，而不是邊搬邊猜問題出在哪。

11 AI Eval

如果你真的要持續細調 LLM / RAG，評測與回歸要怎麼做

真正讓 AI 進步的，不是一直換 prompt，而是建立一個 可反覆比較的 evaluation loop。 沒有固定測題、固定 resume、固定 snapshot、固定 rubric，你很容易把一次看起來比較順的回答誤認成真正進步。

1. 固定資料集

至少準備固定 snapshot、固定 resume 樣本、固定 follow-up conversation。

2. 固定題型

市場摘要、個人化建議、職缺比較、缺履歷 blocked case、無資料 fallback case 都要有。

3. 固定 rubric

不是只看「像不像人講話」，而是看 grounding、coverage、citation、actionability、safety。

4. 固定回歸節奏

每次改 prompt、chunk policy、embedding model、reranker，都跑同一套評測。

評測維度 要看什麼 對 Job Radar 特別重要的原因 建議打分方式
Grounding 回答是否真的由 chunks 支持 求職場景很怕幻覺，尤其薪資、條件、技能缺口 0/1 或 1~5 分；另記 unsupported claims count
Coverage 是否涵蓋問題真正需要的面向 市場摘要常會漏掉技能、來源差異或薪資層面 以子題 checklist 計分
Personalization 有沒有正確用到 resume 與 role targets 個人化 guidance 是這套 assistant 的核心價值 1~5 分，並檢查是否錯引履歷資訊
Citations 引用是否對題、是否能支撐段落結論 citation 不只是裝飾，而是可信度機制 看 top citation relevance 與 chunk type appropriateness
Actionability 建議是否具體到可以行動 求職產品不是只要講得漂亮，而是要能幫使用者下一步 1~5 分，並檢查是否有具體 next actions
Latency / Cost 回答是否在預算內 品質再高，若延遲太高或 token 太貴也很難產品化 記 p50 / p95 latency 與 average token cost

最常見的假進步

  • 只看單題回答，看起來更流暢就以為更好。
  • 換模型後語氣更自然，但 grounding 其實變差。
  • 調大 top-k 後 coverage 變高，但 citation relevance 明顯下降。
  • prompt 變長後看似更完整，但 latency 和 cost 上升太多。

比較健康的調校循環

  • 一次只改一個變數：prompt、chunking、weights、model 其中之一。
  • 每次改完都跑固定 evaluation set。
  • 把質化觀察寫進實驗紀錄，不只留最後版本。
  • 若 quality 進步但 cost / latency 變差，也要記 trade-off，而不是只看答案漂亮度。
我會怎麼幫這個專案建立第一版 AI regression pack？

第一版其實不用很大。我會先準備： 10 題市場摘要、10 題個人化建議、10 題職缺比較、5 題缺履歷 blocked case、5 題 retrieval 容易錯的 follow-up questions； 再搭配 3 份固定 resume profile 與 3 份固定 market snapshots。 每次調 assistant，我只要跑這批固定題，就能快速知道是 mode routing 變差、retrieval 變差、citation 變差，還是 generation 變差。 這會比盲目手問幾題有效得多。

12 Recommendations

如果要把這套 AI 往產品級再推一步，我會先做什麼

不是所有優化都該現在做。以下是依投資報酬率排序，我會建議的下一步。

1. 補手動回歸與評測集

先選一批固定問題與履歷樣本，讓你之後每次調 AI 都能回歸比較。

2. 把監控再拆細

至少把 retrieval、generation、citation selection 的 latency 分開記。

3. 把 assistant 加 second fallback

現在履歷分析 fallback 很完整，但 RAG 助理還可以再補 answer-level fallback。

4. 長期再考慮向量資料庫

等 chunk 量、查詢量上來後再做，現在先把 retrieval policy 定穩比較重要。

一句話總結這套 AI 架構

它本質上是一個 fallback-first 的履歷分析器 + snapshot-centered 的 RAG 市場助理 + event-based 的 AI 觀測層。 這種設計不追求最炫的 agentic 架構，而是優先保證在求職產品場景裡，答案可追、流程可退、成本可控、錯了能查。
