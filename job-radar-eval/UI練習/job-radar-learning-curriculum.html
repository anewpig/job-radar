Learning Map

透過 Job Radar 專案，你可以學到什麼

這份頁面不是介紹功能，而是把這個專案當成教材，拆成一套可學習的課綱。 你可以把它當成「從 side project 走向產品與系統設計」的地圖。每一條學習路線都會對應到這個專案裡真實存在的模組與決策，不是抽象空談。

你能學到的核心能力

  • 怎麼把一個工具重新定義成產品，而不是只會做功能。
  • 怎麼從使用者問題反推功能、狀態與資料邊界。
  • 怎麼做同步 / 非同步流程切分，讓產品真的能跑。
  • 怎麼讓 AI 當 enhancement，而不是變成不穩定的核心依賴。
  • 怎麼拆 DB、拆模組、拆責任，讓系統不失控。
  • 怎麼從 prototype 往可營運產品演進，而不是一直重寫。

適合誰看

PM

學產品定位、JTBD、留存、路線圖、功能優先順序。

工程師

學狀態設計、資料流、模組邊界、AI integration、營運思維。

創作者

學怎麼把 side project 做成能持續長大的產品骨架。

01 Learning Tracks

六條主學習路線

你不一定要照順序全部學，但這六條基本上就是這個專案最有教材價值的地方。

Track A

產品思考

學怎麼從「職缺爬蟲」重新定義成「求職工作台」，理解定位、JTBD、旅程、價值鏈、KPI、留存與產品路線。

定位 JTBD 留存
Track B

系統設計

學怎麼切同步 / 非同步流程、狀態 ownership、模組邊界、查詢身份、背景 worker 與產品 runtime。

queue runtime ownership
Track C

AI 產品化

學 RAG 怎麼接產品、履歷分析怎麼 fallback-first、AI 監控怎麼做、AI 該放在哪一層。

RAG fallback monitoring
Track D

資料與 DB

學多個 SQLite 邊界怎麼切、產品資料與執行資料怎麼分、歷史資料與即時資料怎麼共存。

product state query runtime history
Track E

前端架構

學 UI 不只是版面，而是導航、session state、launcher、頁面 orchestration、section renderer、styles 模組化。

navigation session sections
Track F

重構與營運

學怎麼把大檔拆乾淨、保留相容入口、補 smoke test、做備份、監控、runbook 與維運思維。

refactor backup ops
02 Curriculum

建議課綱順序

如果你要真的把它學進腦子裡，而不是只看熱鬧，我建議照下面順序走。

Lesson 1

先重新定義產品

先回答這個專案到底在賣什麼價值。學會把「抓資料」升級成「幫助求職決策」。

  • 產品定位
  • 核心承諾
  • 使用者旅程
  • 留存機制
Lesson 2

學會切狀態與資料邊界

你會理解為什麼有 session、product store、query runtime、history store，不是所有東西都塞一個地方。

  • state ownership
  • query signature
  • snapshot boundary
  • saved search identity
Lesson 3

學同步 / 非同步設計

你會理解哪些流程該在畫面上等，哪些該交給 scheduler / worker。

  • 初始波查詢
  • 背景刷新
  • queue lease
  • retry / finalize
Lesson 4

學 AI 怎麼放對位置

這個專案可以直接教你 AI 不應該當唯一 source of truth，而是做理解層與增強層。

  • 履歷 fallback-first
  • snapshot-centered RAG
  • AI monitoring
  • token / cache 策略
Lesson 5

學 UI 與頁面分層

你會看到複雜 UI 不是只靠一個頁面大檔撐，而是 orchestration、actions、sections、styles 分開。

  • page orchestration
  • actions / sections
  • navigation / launcher
  • style modules
Lesson 6

學怎麼把專案做成可營運

不是只有能跑，而是知道怎麼備份、怎麼看健康、怎麼分類失敗、怎麼擴充。

  • backup / restore
  • monitoring KPI
  • maintenance
  • roadmap
03 What It Teaches

透過這個專案，你能被教會的東西

下面這張表是最實用的部分。它不是介紹名詞，而是直接告訴你這個專案能教你哪一類真本事。

Job Radar 可教的能力地圖
能力 在這個專案裡怎麼學 你會真正理解什麼 如果學會了能做什麼
產品定位 從搜尋、saved search、履歷分析、AI 助理、看板串起來看整體價值鏈。 工具和產品的差別。 你之後做任何 side project，都更能想清楚真正賣的是什麼。
產品旅程設計 看第一次搜尋、第二次回來、定期刷新、通知、追蹤與看板之間的關聯。 activation、retention、action 不是三個分開的功能。 你能設計更完整的使用者旅程，而不是只做頁面功能。
系統分層 研究 UI、service、store、runtime、scheduler、worker 的邊界。 同一個產品怎麼同時有同步互動與背景工作。 你能設計更穩的中小型產品系統。
資料邊界 看多個 SQLite 與 snapshots 的分工。 不是所有資料都應該放同一個 DB。 你之後做 SaaS、internal tool、AI 產品都比較不會把資料做亂。
AI 產品化 看履歷分析、RAG、fallback、monitoring。 AI 該放在什麼位置才合理。 你可以設計更穩、更可控的 AI 功能，而不是只會 call API。
前端架構 看 navigation、launcher、page actions、section renderers、style modules。 複雜 UI 怎麼維持可讀性。 你能把越做越大的前端保持可維護。
重構能力 這次你前面做的 20 多輪拆檔，本身就是教材。 怎麼拆大檔、保相容、清責任邊界。 你之後接手爛專案時，會比較知道怎麼下刀。
營運思維 看 scheduler、worker、backup、maintenance、notification、health views。 產品做出來後，不是放著就好。 你會開始思考 runbook、告警、備份、恢復與健康度。

05 Practice

如果我真的要教你，我會叫你做哪些練習

學系統設計不是只讀文章。下面這些練習都能直接用這個專案來做。

Practice 1

重寫產品定位

用一句話、一段 50 字說明、一段 150 字說明，重新定義這個專案在賣什麼。

Practice 2

畫 state ownership 圖

把 session、product store、runtime queue、history store、user submissions 的責任畫出來。

Practice 3

拆同步 / 非同步流程

把搜尋、履歷分析、AI 助理、saved search 刷新分成應該同步和應該背景化的部分。

Practice 4

設計 KPI 儀表板

定義 activation、retention、crawl health、AI cost 四種指標，寫出你要看什麼。

Practice 5

設計企業版擴充

想像你要做 team workspace，畫出 user / workspace / role / shared search / shared board 模型。

Practice 6

做 AI fallback 設計

針對履歷分析與助理問答，分別定義 blocked、fallback、error 三種狀態與 UI 表現。

06 Learning Paths

如果你是不同角色，我會怎麼教你

同一個專案，可以教不同角色不同東西。重點是順序不一樣。

PM 路線

先學產品定位、JTBD、旅程、KPI、路線圖，再看系統拓樸。不要一開始就鑽實作細節。

工程師路線

先學 state、資料流、同步 / 非同步，再看 UI 分層與 AI integration。這樣最能把技術和產品對起來。

創業者路線

先學產品定位、留存機制、可營運性、成本與 AI 位置。你最需要的是知道什麼該先做、什麼不該先做。

07 30-Day Plan

如果你要用這個專案練功，建議 30 天學習計畫

這不是唯一方式，但如果你想有效率地吸收，我會這樣排。

Week 1

產品與旅程

看產品手冊、系統總架構頁，重新寫一版產品定位與 JTBD。

Week 2

狀態、資料與 runtime

看 DB 手冊、完整結構圖，自己畫 state ownership 與主要資料流。

Week 3

AI 與 fallback

看 AI 詳細流程頁，整理 resume flow、RAG flow、monitoring flow，自己重講一次。

Week 4

重構與擴充

用這個專案練一次模組拆分、頁面分層與 roadmap 設計，寫出你的下一版產品計畫。

08 Why This Project

為什麼這個專案很適合拿來教你

不是每個專案都適合教學。這個專案特別適合，是因為它剛好卡在「不是太小，也還沒大到失控」的點。

有產品性

不是單點 script，而是一條完整價值鏈。

有系統性

有 UI、service、store、queue、scheduler、worker、AI、DB。

有真實取捨

不是課本範例，而是真的要面對延遲、成本、fallback、營運。

還能演進

它還沒被框架或組織規模鎖死，很適合拿來練產品與架構決策。

一句話總結這份學習地圖

這個專案最能教你的，不是「怎麼寫一個爬蟲」，而是「怎麼把資料、AI、使用者流程和系統邊界組成一個真的能成長的產品」。

這份頁面是學習地圖，適合拿來規劃你接下來要補強的工程、AI、資料與測試能力。
