Scope

AI 在這個專案裡到底做哪些工作

這個專案的 AI 不是只有一條鏈。它至少有四個實際工作面：求職問答、履歷 profile 建構、履歷與職缺語意匹配，以及檢索層的 embedding / persistent ANN。

1. AI 助理問答

由 JobMarketRAGAssistant 驅動。它會先分類回答模式，再決定如何檢索、組 prompt、選 citations；具體職缺薪資題現在還會接薪資預估支線。

  • 主問題：市場摘要、個人化建議、職缺比較、一般對話
  • 核心價值：把當前快照轉成可問答的求職市場語境
  • 風險：回答品質高度依賴 chunking 與 retrieval，不是只靠模型本身

2. 履歷 profile 建構

使用 LLM 將履歷文字抽成 summary / target_roles / core_skills / preferred_tasks / match_keywords，供後續 match 與 personalized guidance 使用。

  • 不是把整份履歷直接丟給回答 prompt，而是先做 profile 化
  • 有 cache，可避免相同履歷反覆抽取
  • 缺點是目前仍偏摘要級，experience / project 尚未 chunk 化

3. 履歷與職缺匹配

這條鏈不是傳統 RAG，而是語意比對。會把履歷與職缺技能/任務/關鍵字轉成 embedding，再計算 semantic similarity。

  • 另有 title similarity LLM 與 rule-based scorer
  • 目前比較像 ranking system，不是聊天式回答
  • 這條鏈對面試很重要，因為它能證明你不只做 chatbot

4. 長期向量與檢索 cache

專案裡不只有 embedding cache，現在本地也已經有 assistant_vector_index.sqlite3。比較準確的描述是：persistent ANN 已能在本地工作，但還不是 shared vector service。

  • local assistant_vector_index.sqlite3：已存在
  • rag_embeddings：主助理熱路徑的磁碟 cache
  • resume_match_embeddings：履歷 matching 的語意 cache
  • 下一步：shared backend、health metrics、multi-instance strategy

Model Stack

模型配置、路由策略與輸出預算

這套系統不是單一模型硬打到底，而是把 assistant、resume extraction、title similarity、embedding 分成不同責任。這是產品級 AI 系統很重要的一點，因為不同任務的延遲、成本、精度需求不同。

Execution Topology

AI / LLM / RAG 執行結構圖

這張圖把「UI 點一下問問題」之後，背後實際發生的事情拆開來看。重點不是把 OpenAI 當唯一中心，而是把資料快照、檢索、prompt、回答、監控分成不同責任層。

1. UI / Session 問題從 `pages_assistant` 或 launcher 送出，並附帶目前 snapshot、resume profile、最近對話上下文。
→
2. Answer Mode Routing 先判斷是 `market_summary / personalized_guidance / job_comparison / general_chat`，再決定 top-k 與是否偏向市場檢索。
→
3. Retrieval Query Build 把本輪問題與最近問答上下文壓成 retrieval query，避免 follow-up 問題失去前文。
4. Chunk Build 從 snapshot 建 `job-summary / job-skills / job-work-content / market-* / resume-summary` 等結構化 chunks。
→
5. Retrieval / Rerank 先粗篩 candidates，再做 embedding，相加 lexical / signal / source bonus / market penalty，最後 diverse top-k。
→
6. Prompt Assembly + LLM 把 answer mode block、style block、question focus、chunks、resume block 一起組成回答 prompt。
7. Structured Response 模型輸出 JSON，再轉成自然段落 QA；UI render 不再硬顯示「重點 / 限制 / 下一步」。
→
8. Monitoring 把事件寫進 `ai_monitoring_events`，目前有 mode 與 latency，但 token usage 尚未穩定落地。
→
9. Product Memory 問題、回答與對話上下文留在 app session / product state，支撐 follow-up 與 personalized guidance。

回答模式、prompt 策略與目前的設計問題

現在 answer mode 兼管了三件事：題目類型、回答口吻、某種程度的 retrieval policy。這讓系統能跑，但也是你後面最需要重構的一塊。

目前四種 answer mode
Mode 主要用途 目前風格 是否偏向市場檢索 風險
market_summary 技能、工作內容、來源、薪資、市場走勢 正式 QA 幾乎必用 太多責任被塞在一起，容易把市場題和 job-specific 題混在同一框架
personalized_guidance 根據履歷給缺口與下一步 正式 QA + 履歷參照 必用 履歷目前還是單一 summary chunk，個人化粒度偏粗
job_comparison 比較兩個職缺的技能、內容、薪資、風險 正式比較 必用 比較題目前靠 `_ensure_comparison_coverage` 補位，之後可再有專用 rerank

Resume AI

履歷 AI 鏈不是 RAG 附屬，而是另一條獨立能力線

很多人會把這個專案只講成 AI 助理，但其實履歷分析是另一條完整 AI 鏈：先抽 profile，再做 resume-job matching，再把結果回寫成可視化分析。這條線更能證明你不是只會做 chatbot。

1. Resume Upload 使用者上傳履歷，系統先抽文字，再做清理與上下文選取。
→
2. Profile Extraction 用 LLM 抽出 `summary / target_roles / core_skills / preferred_tasks / match_keywords`。
→
3. Resume Chunk / Facet Build 把 profile facets 與 `resume-window-*` 變成後續 personalized guidance 的上下文層。
4. Market Job Pool 直接吃目前 `jobs_latest.json / snapshots`，以當前市場職缺池為匹配基準。
→
5. Matching / Scoring 結合 exact match、semantic similarity、title similarity、keyword/task/skill scorer。
→
6. UI Analysis 輸出缺口技能、推薦職缺、匹配分數與摘要解釋。

這條鏈目前已做好的

  • 履歷抽取有 cache，不會每次都重做相同 profile
  • matching 不只靠 embedding，還有 exact skill / task / keyword / role signal
  • 履歷已從單塊 summary 升級成多 facet + windows
  • resume.service、extractors、matchers、scoring 責任已拆乾淨

這條鏈還缺的

  • experience / project / accomplishment 級別的 evidence chunks
  • 更正式的 profile extraction answer-level eval
  • resume retrieval 與 market retrieval 的雙路融合策略
  • 可視化顯示「這段履歷證據對應到哪些職缺要求」

Retrieval

Retrieval / Rerank / top-k 目前怎麼跑

這套檢索不是只用向量相似度，而是完整 hybrid retrieval。這是你在面試時應該主動強調的地方，因為它代表你知道「只上 embedding 不等於產品級檢索」。

final_score = embedding_score * 0.52 + lexical_overlap * 0.18 + signal_overlap * 0.22 + source_type_bonus + market_priority_bonus + role_alignment_bonus + job_reference_bonus - specific_job_market_penalty

top-k routing

  • specific job + `work_content / skill_gap / salary`：5
  • `market_summary`：6
  • `personalized_guidance`：6
  • `job_comparison`：10
  • `general_chat`：4

這代表系統已經不是固定 `top_k=8`。你現在會依題型壓縮 context，避免市場題、一般題與比較題都吃同一套上下文預算。

目前 rerank 的關鍵意圖

  • `work_content` 題會強拉 `job-work-content`
  • `skill_gap` 題會強拉 `job-skills / market-skill-insight`
  • specific job 題會對完整 title / company 追加 bonus
  • specific job 題時，market chunk 會吃 penalty，避免市場摘要壓過單一職缺
  • 最後還會做 diversity selection，避免 top-k 全被同類 chunks 佔滿
為什麼這套 retrieval 對產品有意義
設計 為什麼這樣做 優點 缺點 / 成本
先 coarse candidate，再 embed 避免每題對全庫所有 chunks 做 embedding 比分 速度與成本比較穩 coarse stage 設計不好會一開始就把正確 chunk 篩掉
hybrid score 純 embedding 不足以表達 job-specific / market-specific 的產品意圖 對結構化資料更友善 rule 會越堆越多，維護成本上升
specific job bonus 解決「問某個缺，卻被市場摘要搶走」的常見錯誤 命中率明顯上升 title alias / company alias 還不夠完整時，仍會有誤差
diversity selection 避免同一類 chunks 佔滿 top-k 讓 prompt context 更均衡 如果 diversity 規則太硬，也可能稀釋最強證據
ANN / Vector Index

長期持久化向量索引目前做到哪裡

專案裡的 PersistentANNIndex 已不是只有空骨架。現在本地 data/assistant_vector_index.sqlite3 已實際存在， 而且程式路徑能同步 runtime snapshot / snapshot_file / snapshot_store / market_history。目前比較準確的說法是： 這塊已經進入「可工作的本地 persistent ANN」階段，但還不是多實例共享、獨立向量服務化的 production 最終形態。

目前已具備

  • SQLite-backed persistent vector corpus
  • random-projection LSH band search
  • 可同步 runtime snapshot
  • 可同步 snapshot file / snapshot store
  • 可同步 market_history job_posts
  • runtime snapshot sync 已改成 hash-based incremental style
  • 本地已觀察到 assistant_vector_index.sqlite3 實體檔案，表示 steady-state index 不再只是理論 code path

目前仍然要誠實講的限制

  • 現在是 product-local SQLite index，不是 shared vector service
  • 還沒有外部 dashboard 去追 corpus growth、fan-out、recall 趨勢
  • 多實例上線時不能直接把這一顆本地 SQLite 當共享索引
  • 若未來 corpus 持續擴大，仍建議升級成獨立索引服務或共享 backend
這一段面試時可以這樣講：我不是只停在 embedding cache，而是已把 persistent ANN index 路徑落地到本地可運作狀態；下一步要補的是 shared backend、observability 與多實例部署策略。
Embedding / Cache Layer

Embedding、cache 與向量層在系統裡的實際角色

目前預設 embedding model 是 text-embedding-3-large。系統不是把所有原始資料直接丟去做 embedding， 而是先把 snapshot / resume 轉成結構化 chunks，再把 retrieval query 與 candidate chunks 一起向量化。也就是說， embedding 在這裡負責「先找語意近的候選」，最後排名仍要靠 hybrid score 收斂。

目前 embedding / cache 分層
層 目前用途 資料位置 目前優點 目前缺口
process memory cache 降低同一輪互動重複 embedding / profile / title 計算 app process memory 對熱路徑延遲很有效 重啟後會消失，沒有跨 process reuse
disk cache 保存 assistant / resume embeddings 與中間結果 data/cache/rag_embeddings、其他 resume caches 成本低、易維護，而且 repeated question 不必每次都打遠端 embedding API 仍不是正式的 metrics backend，也缺 cache hit 長期趨勢圖
persistent ANN index 做長期候選召回 data/assistant_vector_index.sqlite3 已能承接 runtime / snapshot / history 三種來源 還不是 shared backend，也未拆成獨立索引服務
monitoring metadata 追 cache hits、remote embeddings、ANN fan-out ai_monitoring_events 已能回看是哪一層 cache 沒打中 還沒做長期 trend dashboard 與告警

實際被拿去 embed 的不是什麼

  • 不是整份 HTML 原文。
  • 不是整個 JSON snapshot 一次送進去。
  • 而是 job-summary / job-salary / job-skills / job-work-content / market-* / resume-* 這些結構化 chunk。
  • query 也不是只用原句，而是會帶上 answer mode 與最多 2 輪對話摘要。

這條路徑為什麼不是純 cosine similarity

  • embedding 先把語意近的候選拉出來。
  • lexical overlap / signal overlap 補字面與任務訊號。
  • source type bonus 把對的資料型態往前推。
  • specific-job 問題還會壓低 market chunks，避免「語意像但類型錯」的證據搶位。
這條鏈真正的意思是：embedding 在系統裡不是單一功能，而是 retrieval、matching、latency control、cost control 的共同基礎層。

薪資預測 v1：Hybrid RAG + Interval Estimation

AI 助理現在除了傳統 RAG 問答，還多了一條「具體職缺薪資題」支線。這一版不是做全市場盲猜，而是只在 具體職缺、原始職缺沒有真實薪資 的情況下，回傳 AI 預估月薪區間 + 相似職缺證據 + 限制說明。

產品行為

  • 若 job card 或 assistant 指到的職缺已經有真實薪資，就直接優先顯示真實薪資。
  • 若是具體職缺薪資題，且職缺沒有真實薪資，就走薪資預估支線。
  • 若是「AI 工程師市場薪資大概多少」這種市場題，仍維持 RAG-only，不會硬做泛化預測。
  • 若模型檔不存在、信心不足、或特徵不足，assistant 會回退到純 RAG，job card 則不顯示 AI 預估 chip。

模型與證據層

  • baseline 模型走 TF-IDF + OneHot + numeric features + Ridge low/high regressor。
  • 訓練資料來自 jobs_latest.json 與 market_history.sqlite3 的可解析薪資職缺。
  • 預測後不是只丟一組數字，而是再檢索 2-3 筆相似且有揭露薪資的職缺作 evidence。
  • 這條線是 Hybrid RAG + prediction，不是單純 LLM 自由生成，也不是純回歸黑盒。

離線 benchmark 與目前暴露出來的真問題

目前專案已有兩類 retrieval benchmark：一組偏標準題、一組偏真實提問。前者已飽和，後者才真正能反映現在的短板。這一輪又再往前走一步，把 prompt variant 與 retrieval baseline 綁進 release gate，讓 benchmark 不只是觀察工具，而是放版門檻的一部分。

Benchmark 設計

Gaps & Improvements

目前最值得改善的點

下面這些不是「想得到但還沒做」的 wishlist，而是根據現有 benchmark、live event 與結構觀察後，最值得優先處理的項目。

1. `work_content` 題仍容易被市場技能 chunk 搶走

這是現在最明確的 retrieval 問題。雖然你已經加了 specific job bonus 與 market penalty，但 realistic set 的 miss samples 仍顯示 `主要在做什麼` 類題目，前幾名常出現 `market-skill-insight`。

  • 改善方向：把 `work_content` 題的 coarse candidate score 再往 `job-work-content` 傾斜
  • 改善方向：把 market insight 與 raw job detail 分 namespace 搜尋
  • 改善方向：對 `做什麼 / 職責 / 負責什麼` 類題單獨加 reranker

2. mode 設計仍混著題型、口吻、retrieval policy

現在 `market_summary` 和 `general_chat` 的問題，不是 prompt 寫得差，而是 mode 名稱本身已經承載太多責任。這會讓 debug、ablation 與 prompt 演進越來越難。

  • 改善方向：拆成 `subject_mode / tone_mode / retrieval_mode` 三層
  • 改善方向：讓 general chat 真的只管語氣，不要兼任檢索策略

3. 本地向量索引已存在，但 shared corpus 還未成熟

現在本地已經能看到 assistant_vector_index.sqlite3，代表 persistent ANN 不再只是空的 code path。不過它目前仍是 product-local SQLite index， 不是多實例共享的索引服務，因此「有本地 index」不等於「向量基礎設施已經全部完成」。

  • 改善方向：補 index health / corpus size / recall 監控
  • 改善方向：把本地 SQLite index 升級成 shared backend 或獨立索引服務

4. 延遲優化已上線，但還缺 steady-state 對照報表

這輪已經把 chunk cache、runtime ANN sync、resume context、output budget 都往下收，而且 backend console 也已能看 cache hit / remote embedding / prompt version 分布；但目前 live `ai_monitoring_events` 還沒有一組「優化前 / 優化後」可直接對照的 steady-state 報表。

  • 改善方向：把 deployment time / code version / tuning profile 寫進 monitoring metadata
  • 改善方向：把 usage tokens、retrieved source types、citation count、retrieval mode 一起落監控表
  • 改善方向：補一張優化前後 latency 對照報表，而不是只看單點平均值

5. 履歷 personalization 的 chunk 粒度仍太粗

這一輪已經把履歷從 `resume-summary` 單塊升級成 facets + `resume-window-*`，所以 personalized guidance 的基礎比前一版好很多；但它仍然還不是 project-level / experience-level 的精細證據層。

  • 改善方向：拆 experience / project / accomplishment / tool evidence chunks
  • 改善方向：把 resume retrieval 與 market retrieval 做雙路融合

6. Prompt A/B 已有版本治理，但 answer-quality gate 還沒補齊

現在 prompt 已有 `control / compact_qa` 變體、也有 version registry，而且 CI / release 已接 retrieval regression gate。下一步要補的是「模型最後回答有沒有更好」，而不只是 retrieval 有沒有退步。

  • 改善方向：補 answer-level review set，量摘要正確性、引用對齊、篇幅控制
  • 改善方向：把 `prompt_variant` 的線上事件分布做成 A/B 看板

7. Snapshot lineage 已落 metadata，但還沒有獨立品質儀表板

snapshot 現在已經會帶 `snapshot_kind / schema_version / data_quality / lineage / snapshot_query_signature`，對 debug 跨來源 merge 非常有幫助。不過這些資訊目前還主要活在 snapshot payload 裡，還沒被整理成單獨 dashboard。

  • 改善方向：做 `source_record_id / canonical_identity_key / lineage trail` 查詢頁
  • 改善方向：把 dedupe rate / merge rate / detail enrichment rate 做成固定報表

Interview Script

面試時怎麼講這套 AI / LLM / RAG

這一段不是技術文件，而是講法。你最好不要只背功能列表，而是用「產品問題 -> AI 放在哪裡 -> 現在做到哪裡 -> 哪裡還沒做完」的順序講。

5 分鐘版本

  • 這個產品不是純爬蟲，我把它做成求職工作台，AI 只是其中一層。
  • AI 助理不是直接把所有資料丟給模型，而是先有市場快照，再把職缺切成結構化 chunks 做 hybrid retrieval。
  • 模型不是唯一核心，我同時做了 chunking、top-k routing、rerank bonus / penalty、離線 benchmark。
  • 目前 production realistic retrieval 還有 headroom，工作內容題最容易被市場技能 chunk 搶走，這是下一步會優先修的點。
  • 我也已經把長期向量索引的程式骨架落地，但 live corpus 還在收尾，這是我後續會補的基礎設施。

10 分鐘版本

  • 先講產品：使用者真正要的是「快速理解這波職缺在要什麼」，不是單純抓資料。
  • 再講 AI 切面：assistant 問答、resume profile、resume-job matching、embedding cache / vector index 是四條不同能力，不混為一談。
  • 再講 retrieval：我不是只做 semantic search，而是 embedding + lexical + signal + source bonus + specific-job penalty 的 hybrid rank。
  • 再講數據：default benchmark 已飽和，但 realistic set 仍只有 78.95% hit@1；問題集中在工作內容題。這代表我有量測，也知道真正弱點在哪。
  • 最後講 roadmap：先修 retrieval 與 resume chunking，再把 ANN index 與 observability 完整接起來。
面試時最好的姿態不是把它講成「我都做完了」，而是講成：我知道這套系統現在已經到什麼成熟度、最真實的缺口在哪、下一步怎麼用最低成本把準確度拉高。
2026-04-12 Update

AI 助理已從單純 RAG assistant 升級成求職工作流 agent

這一輪沒有把既有 RAG 全部推倒重寫，而是在既有 `classify → retrieval → answer/report` 上面再包一層 orchestration。現在 AI 助理不只能回答問題，還會先進入 planning phase，先把任務拆成 plan steps；等使用者按下執行後，才進入 execution phase，逐步調用 tools，並在寫入型動作前停下來等使用者確認。

這次實際改了什麼

  • 新增 `JobSearchWorkflowAgent`，先判斷輸入要走 `assistant` 還是 `agent` 路徑。
  • 新增 `AgentIntent / AgentTaskPlan / AgentStep / AgentToolCall / AgentExecutionResult / AgentPendingConfirmation` 這組內部 schema。
  • 互動流程已明確拆成 `plan → execute → confirmation`，送出任務後先只顯示計畫，不會立刻動系統狀態。
  • read tools 現在可直接執行：`inspect_snapshot`、`start_or_refresh_search`、`summarize_market_snapshot`、`generate_job_report`、`switch_main_tab`、`open_relevant_surface`。
  • write tools 現在一定先過 confirmation gate：`save_assistant_profile`、`create_or_update_saved_search`、`update_notification_preferences`。
  • 如果更新個人背景時同時改了 `target role`，確認寫入後會順便刷新搜尋快照，不再只改 profile 卻保留舊職缺結果。
  • AI 助理頁的個人化背景卡現在改成「履歷摘要 + 可補充表單」模式；即使已載入履歷，使用者仍可補充目標職缺、技能與地點，而 assistant 會真正使用合併後的 profile。
  • UI 不另外做新 console，而是沿用原本 assistant page，用「聊天 + 任務卡」顯示計畫、結果、待確認寫入。

為什麼這樣拆，而不是直接做通用 agent

  • 這個產品的任務邊界很清楚，主要就是求職搜尋、報告、saved search、通知、個人背景這幾類工作流。
  • 底層 RAG 已經能回答市場問題，所以不需要把 retrieval / prompt 邏輯混進 agent；agent 只做 orchestration。
  • 先把 tools 固定在產品內已存在能力上，比直接開放通用 function calling 更可控，也更好測。
  • 寫入動作全部先確認，代表它還是 assistant-first，而不是自治型 agent，這比較符合求職場景。
  • tab 切換現在也只在明確出現「切到 / 打開 / 開啟」時才觸發，避免把「技能」這類一般描述誤判成切頁指令。

執行流程

  • 1. 使用者提問。
  • 2. agent router 判斷這題是一般問答、RAG 問答，還是 workflow task。
  • 3. 若走 agent，先只產生 plan steps，讓使用者先看到這次打算怎麼做。
  • 4. 使用者按下「開始執行計畫」後，executor 才逐步執行 tools。
  • 5. 只要遇到 write tool，就先產生變更摘要卡，等待使用者確認。
  • 6. 確認後才真的寫入，最後回傳結果摘要與下一個建議入口。

這樣做的優點

  • 把「回答」和「動作」分開，避免本來只想問問題卻被直接改設定。
  • 既有 RAG assistant 不用重寫，風險比整顆改成 agent 小很多。
  • 任務卡可直接展示給使用者看，也很適合面試時講系統演進。
  • read / write tools 有清楚邊界，之後要補 multi-agent 或 background jobs 也比較好接。

第一版仍刻意保留的限制

  • 只處理求職工作流，不做通用 agent。
  • 一個使用者一次只跑一個 agent task，不做多任務並行。
  • guest user 不允許持久化寫入。
  • agent 不碰 backend console、database 與 operator-only surface。
  • 背景自治排程 agent 也還沒開，這一版是同步互動式 agent。
Agent orchestration 現在怎麼對接既有 AI / RAG
層級 現在的責任 和原本系統的關係

這次最值得講的，不是「我把 AI 助理變得更聰明」，而是 我把它從只會回答問題的 RAG assistant，推進成可規劃、可執行、可確認、可稽核的 job workflow agent。這是產品成熟度很明顯的一步。
2026-04-11 Update

觀測、版本治理與延遲控制新狀態

這輪不是再調回答文案，而是把 AI / LLM / RAG 這條線往 production hardening 推：補上 trace、prompt/version registry、cache efficiency 指標、backend status build/schema 報表，以及更清楚的 deployment / release 治理骨架。

新增的 runtime metadata

  • assistant 事件現在會記錄 `selected_model`、`answer_mode`、`top_k`、`chunk_cache_hit`、`persistent_ann_candidates`。
  • retrieval 事件會記錄 `candidate_chunk_count`、`retrieval_scored_chunk_count`、`retrieved_chunk_count`。
  • embedding 路徑會記錄 `embedding_memory_hits / embedding_disk_hits / embedding_remote_texts / embedding_remote_batches`。
  • resume 路徑會記錄 `profile_cache_memory_hit / profile_cache_disk_hit / profile_cache_write`、`title_cache_memory_hits / title_cache_disk_hits`。
  • assistant / resume 監控事件都已加上 `trace_id` 與標準化 `error_kind`。
  • assistant 事件也會記錄 `prompt_variant` 與 `prompt_version`，方便之後做 A/B 與 release 對照。

這輪真正補到的治理能力

  • build info 已顯性化：`package_version / api_version / deploy_env / release_channel / git_sha`。
  • schema registry 已顯性化：`product_state / query_runtime / market_history / auth / audit / prompt_registry`。
  • backend status 報表現在會帶 AI latency budgets、cache efficiency 與 recent audit event 概況。
  • logging 已支援 `text / json` 兩種格式，且可在 log 上直接看到 `trace_id`。
  • CI / release workflow、changelog、deployment runbook 已補進 repo，讓 prompt / schema / AI tuning 能納入 release discipline。
  • snapshot payload 現在會帶 `snapshot_kind / schema_version / data_quality / lineage / snapshot_query_signature`，AI 之後可直接追到資料來路。

這輪真正新增的 AI / RAG 能力

  • prompt variant 已落地，支援 `control / compact_qa`，並綁版本 registry。
  • benchmark 現在不只是報表，CI / release 會用 `realistic` set baseline 做 regression gate。
  • backend console 已新增 `AI Cache / ANN 效率` 與 `Prompt / Retrieval / Model 版本分布`。
  • resume chunking 第一階段已上線，不再只有 `resume-summary` 單塊。
  • snapshot 已補 lineage 與 schema version，讓 AI context 的 provenance 可追。

現在仍然要誠實講的限制

  • release gate 目前量的是 retrieval，不是最終 answer quality。
  • cache dashboard 已進 backend console，但還不是外部 metrics backend。
  • resume chunking 仍未做到 `experience / project / accomplishment` 結構層。
  • snapshot lineage 已在 payload 裡，但尚未整理成獨立 data quality dashboard。
這輪新增的 AI / RAG 可觀測欄位
欄位 用途 你面試時可以怎麼講
`trace_id` 把 UI 互動、AI 監控、log 事件串成同一條請求鏈。 我不是只看平均延遲，而是能追同一次互動從前端到 LLM 的路徑。
`selected_model` 把 mode-based model routing 顯性化，分辨正式 QA 與 general chat 實際用了哪個模型。 我後面如果做 model routing A/B，不需要重新猜每次用的是哪個模型。
`prompt_variant` 把 prompt A/B 與 release baseline 對齊，知道現在線上到底用哪個語氣 / 篇幅策略。 這讓 prompt 調整不再只是文案微調，而是可以進版控與 regression gate。
`prompt_version / retrieval_policy_version / chunking_policy_version / persistent_index_version` 讓 AI tuning 與 release 之間有版本對照。 之後如果 benchmark 漂移，我可以直接對照是哪一版 prompt 或 retrieval policy 造成的。
`embedding_* hits` 把 cache hit 與 remote embedding 成本拆出來看。 延遲不是只看整體平均值，我也能看是哪一層 cache 沒打中。
`profile/title cache hits` 觀察履歷抽取與 title-similarity 這兩條 AI 副路徑是否有重複成本。 resume 不只是單次分析，我已經把 extraction 與 matching 拆開監控。
`snapshot_kind / schema_version / data_quality / lineage` 讓 AI / resume 吃的市場上下文具備版本與來源可追性。 如果某次 merge 或來源品質出問題，我可以從 snapshot 直接往回追資料來路。
這輪最關鍵的提升不是「又多做了一個 benchmark」，而是讓 AI tuning 可以被版本化、被追蹤、被回溯。這會把系統從 demo 進一步推向可維運產品。
Failure Modes

AI / LLM / RAG 最容易出問題的點與現在的應對

完整報告不只要講做了什麼，還要能講哪裡最容易壞、目前如何觀測、還有什麼沒補完。這一段是你面試時很加分的地方，因為它顯示你不是只會堆功能。

主要風險點
風險點 現象 目前應對 仍待補強
retrieval 命中錯誤 chunk 問某個缺在做什麼，卻先回市場技能摘要 specific job bonus、market penalty、top-k routing、benchmark realistic set work_content 專用 reranker / namespace split
OpenAI 依賴延遲或失敗 assistant / resume 變慢或失敗 cache、token budget、error_kind、latency monitoring 更完整 alerting 與 fallback policy
prompt 調整造成品質回歸 回答風格或 grounding 變差 prompt version registry、retrieval regression gate answer-quality gate / A-B 統計
resume personalization 不夠細 建議太 generic，不像真的顧問 facets + resume windows 已落地 experience / project evidence chunks
lineage 不透明 難追某次 snapshot 為何 merge 成那樣 snapshot `data_quality / lineage / schema_version` 已落 payload 獨立 data quality dashboard
